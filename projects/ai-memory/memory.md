## What It Is
The markdown-only memory system for Claude Code — no vector DB, no daemons, no MCP server, just files + hooks + macOS-`bash`-3.2 shell scripts. This is a **meta-project**: the project being tracked *is* the memory tooling. Single owner (Seyi). Authoritative spec is the repo `README.md` (a lean landing page) plus the deep-reference pages under `docs/` — `docs/harnesses/<name>.md` per harness, mirroring the multi-harness direction.

Three layers (Karpathy LLM-Wiki pattern):
- **Schema** — `identity.md` (hard rules, injected once/session), `~/.claude/CLAUDE.md`. Outranks everything.
- **Wiki** — `domain/*.md` (cross-project) + `projects/*/memory.md` (engagement-specific). Durable.
- **Scratchpad** — `projects/*/working.md`, injected with the full payload at session start (re-injected on `@memory`/after compaction) while non-empty — **not** every prompt; matures into the wiki via `/promote-memory`.

Moving parts: one `UserPromptSubmit` hook (the shared `scripts/hooks/inject.sh`, format-parameterized) emits `<memory:*>` blocks; slash commands in `~/.claude/commands/`; scripts in `scripts/` (`_lib.sh`, `regenerate-index.sh`, `lint-memory.sh`, `archive-cleanup.sh`, `new-project.sh`, `memory-pin.sh`, `link-skills.sh`, `executor.sh`, `codex-mem.sh`, the `taskprovider/` package + `taskctl`) with dependency-free bash-3.2 tests under `scripts/tests/`.

## Current State
Functional and in daily use across the real engagements indexed in `index.md`. This checkout is the distributable **source copy** and the dev instance: `origin = seyio91/ai-memory-system` (private, headed public), `AI_MEMORY_CHANNEL=dev` so a sync never flips it onto a tag.

The engine is **harness-agnostic** (`install.sh` is a generic manifest-driven installer; `claude`/`codex`/`antigravity` are registered harnesses; the marker is `.agents/memory-project`; the executor resolves roles from the manifest) and **versioned**: consumer instances sync to stable `v*` tags, never to a moving `main`. Latest tag `v1.1.0`; `v1.0.0` is a trap tag (see Gotchas). Design records live in Architecture Decisions below.

## Architecture Decisions
- **Markdown over a DB** — every editor/diff/grep works; zero infra; git-trackable.
- **Hook-injected, not retrieved** — context arrives in-band. Full payload (identity → project memory → index → working) is injected at SessionStart, on `@memory`, and on the first prompt after compaction; every other prompt gets only the lightweight `<memory:active>` breadcrumb. Domain files are the exception: lazy-read on index trigger-match, never auto-injected.
- **Frontmatter-driven catalog** — `index.md` AUTOGEN block is regenerated from frontmatter (`/reindex`), so it can't lag the files. **Exactly one canonical fence:** `<!-- BEGIN AUTOGEN -->` / `<!-- END AUTOGEN -->` (a `## Projects` roster + a `## Domain` table). The older single combined-table `<!-- AUTOGEN:START -->` / `<!-- AUTOGEN:END -->` block is **retired** — `regenerate-index.sh` never touches it, so any surviving copy silently rots and duplicates the live tables; instances carrying it must delete it. Both scaffold files are excluded from the catalog: the projects loop skips `projects/_template/`, and the **domain loop skips `domain/_template.md`** (its literal `topic: <topic>` frontmatter otherwise leaks a placeholder row) — mirror both guards in any regenerator change, and guard the domain exclusion with a test.
- **Lean index** — `index.md` is a path-less roster; paths are derivable (`projects/<name>/memory.md`, `domain/<topic>.md`), metadata (`tags`/`repo`/`repo_path`, domain `triggers`) stays in the source files. Codex's `AGENTS.md` keeps absolute paths (its shell tool reads by path).
- **No per-session marker; `SessionStart` is the once-per-session guarantee** — the full payload is emitted inline by the `SessionStart` hook, which fires once per session by definition, so nothing needs tracking and concurrent sessions cannot collide (no shared state on the hot path). The only per-session artifact is the post-compaction sentinel `<session_id>.recompact` under `$MEMORY_DIR/.sessions` (override `MEMORY_STATE_DIR`), pruned after 2 days for sessions that compacted but never resumed.
- **Compaction recovery** — `PreCompact` stdout can't inject; the reliable channel is `UserPromptSubmit`. `SessionStart source=compact` writes a per-session sentinel; the next prompt re-injects the full payload once and clears it.
- **Project detection per-prompt** — the hook reads `cwd`, walks up for a harness-neutral `.agents/memory-project` marker (legacy `.claude/memory-project` still read as a fallback at the same level); no marker → no project (memory dormant, no global fallback, so concurrent repos never collide). `.active_project` machinery still exists but is left empty by deliberate decision.
- **Bidirectional repo↔project map** — forward marker `.agents/memory-project` (harness-neutral; `memory-pin.sh` migrates a legacy `.claude` one on pin; `scripts/migrate-marker.sh` bulk-migrates) + reverse `repo`/`repo_path` frontmatter, both written by `/pin` (`memory-pin.sh`). `resolve_repo_path` resolves a checkout per-environment: `repo_path` relative to `projects_root()` (`AI_MEMORY_PROJECTS_ROOT`, default `$HOME/Projects`), git-remote `repo` as portable fallback. `lint-memory.sh` catches reverse-map drift. **`repo_path` is stored relative, never absolute** — the env-var root supplies the prefix so paths resolve identically across host and sandbox. **Self-referential exception:** the `ai-memory` meta-project's checkout *is* the memory tree, so its `repo_path` is the sentinel `$MEMORY_DIR` (and `$MEMORY_DIR/<sub>` is supported) — `resolve_repo_path` and `lint-memory.sh` resolve it against `MEMORY_DIR` rather than `projects_root()`, keeping the committed value machine-independent (each machine sets `MEMORY_DIR` to wherever its tree lives).
- **Distributed cross-project relationships** — a relationship lives in the project where the work *starts* (`## Related Projects`), never in an umbrella. Siblings are **delegated to subagents, never loaded** into the orchestrator thread; plan-set execution pauses at human/CI gates.
- **Orchestrator / Executor / Validator workflow** — Claude main session = orchestrator; an executor runs delegated work; the validator is its **own read-only role** (`AI_MEMORY_EXECUTOR_VALIDATE`, resolved via `executor.sh --role validate`) that **defaults to the orchestrator's agent plane (`claude-subagent`), not the executor's value** — so validation is **cross-model by default** (a CLI executor like codex is checked by a decorrelated model), decorrelating reasoning blind spots and not just context. It resolves through the harness's `exec_readonly` face (degrades to the subagent plane when absent) so a validator verifies but never repairs; independence still also comes from the separate, fresh invocation against the plan's `## Success criteria`. No enforced capability floor — the default just can't self-select a weak model (`claude-subagent` has no `:model` suffix); a weak validator only happens if configured by hand. Harness `TaskCreate`/`TaskUpdate` are banned (enforced by `block_task_tools.sh`); `todo.md` is the single source of truth. Archive is never read unless asked. Executors are blocked from any apply/merge/destructive action on running infrastructure.
- **Executor: two roles, manifest-resolved.** `scripts/executor.sh` exposes `--role task` (write-capable) and `--role explore` (read-only), each configured as a `harness[:model]` value (`AI_MEMORY_EXECUTOR_TASK` / `AI_MEMORY_EXECUTOR_EXPLORE`, falling back to the legacy single `AI_MEMORY_EXECUTOR`, then `claude-subagent`). A harness resolves **through its `harnesses/<name>/manifest` `exec_*` block** — no per-harness code (the old codex special-case is gone): `exec=subagent` → subagent plane; else `exec_cmd` (task) / `exec_readonly` (explore), gated on `exec_probe`. A harness with no read-only mode is **skipped for `explore`** (degrades to the Claude Explore agent), never run write-capable. Model threads via `exec_model_flag`; the subagent plane surfaces `subagent:<model>`. `--which` prints `subagent[:model]` | `cli:<name>`; `--run` execs or bounces to the subagent (exit 3). **Which executor a given instance uses is config-driven, never recorded here** (set the vars in gitignored `config.local.sh`; the orchestrator discovers via `executor.sh --role <r> --which`). The codex deny-rules file is optional codex-only hardening; the no-merge deny-list is restated in every delegation prompt regardless of executor.
- **Task-provider layer + `/task`/`/start`** — pluggable capture/track/execute subsystem; stdlib-only Python `scripts/taskprovider/` (generic registry: env value `MEMORY_TASK_PROVIDER` *is* the provider module → no factory edit to add a backend), `providers/local.py` (flat `tasks/<slug>.md`) + `providers/notion.py` (urllib-only). Backend is a projection + intake queue, push-dominant, never bidirectional. `scripts/taskctl` bash wrapper; `/task` (add|list|done|archive|show), `/start <ref>` (runs the brainstorm gate → scaffolds a plan into the task's own project).
- **Task `summary` is a capped thin record; long-form lives in the tree by pointer** — `summary` is hard-capped at **500 chars**, enforced on write by `validate_summary()` at the `TaskProvider` ABC boundary (applied via `__init_subclass__` to `capture`/`update`, exactly as `validate_status` guards `set_status`). The cap is **backend-neutral by design**: it descends from the projection model, *not* from Notion's 2000-char-per-`rich_text`-element limit — so a `local`-only task obeys it, it fires **before provider dispatch** (an over-cap Notion capture never issues HTTP), and a new provider cannot forget it. **Reads are never gated** — pre-cap tasks keep loading and are corrected on next write; no migration. Long-form goes to `projects/<project>/investigations/<slug>.md`, referenced from the summary **by name (`<slug>`), never by path** — the task record already carries its `project`, and a path rots the moment the work is archived (see the name-not-path decision above). The ValueError text is the primary doc surface. Consequently the `## The model` invariant is narrowed to "**no plan, no `todo.md` row, no index entry** before `/start`" — an investigation *may* exist pre-start, being a design input rather than live work. **Rejected:** chunking `_text_value` across elements (fixes the symptom, keeps the disease — and is moot at ≤500 chars, which always fits one element); a bidirectional `body` field (a push path for long-form into a store defined as a projection). **`add_progress` stays the documented "designed, not wired" no-op** — so no Notion block writer exists, and none is needed until `/checkpoint` is wired. **The Notion page *body* is outside the contract — no reader, no writer** (decided 2026-07-09): the provider touches properties only, so anything typed into a task page's body is silently ignored. This is *by design*, not a deferred gap — reading it would make the page a second home for detail, the split-brain the projection model exists to prevent. (The gap is **read**-side; declining to build the `body` *writer* does not close it — the projection model does.) Documented in the Notion schema section so a human doesn't type a design there and watch it vanish.
- **Brainstorming skill (gated Tier-3 design pass)** — runs the collaborative design pass only for Tier-3 feature tasks with open design questions; silent on Tier 1/2 and settled Tier-3. Terminal handoff is `/new-plan`, folding the approved design into the plan's Goal/Success criteria/Design/Risks. Orchestrator-only, no separate committed spec.
- **An investigation is an artifact; a brainstorm is an activity** (decided 2026-07-10) — the two were conflated under one directory name, `brainstorms/`, and the doctrine split in half as a result: `memory.md` said long-form belongs in `brainstorms/<slug>.md`, while the brainstorming skill said "do not create a separate spec document — the design lives in the plan file, in the memory tree." Both were right about *different files*. An **investigation** (`projects/<project>/investigations/<slug>.md`) is the findings artifact written while exploring, **before a task exists**; it is **conditional** — it exists only when an investigation was actually done *and* its output exceeds the 500-char summary cap. It has two readers: the task summary names it, and `/start` hands it to the brainstorming skill as the **seed**. The brainstorm is the *dialogue*; its only output is the **plan** (`## Goal` / `## Success criteria` / `## Design` / `## Risks`) — never a sibling file. The tell that the name was wrong: `release-automation.md` sat in `brainstorms/` carrying `kind: brainstorm-input` and `status: open (investigation only)`, and its task summary said "`/start` should skip the brainstorm gate." A directory whose contents declare they are not what the directory is named is a naming bug, not a filing error. Generalises to the drift rule in `domain/preventing-drift.md`: **one fact, two homes, and they disagree** — here the fact was "where does long-form live," and the second home was a remote skill in another repo (`agent-skills`), so nothing in this tree could have caught the contradiction.
- **Task Contract — success criteria** — plan-tier tasks carry explicit checkable success criteria (the Validator's contract). Policy in `identity.md` → `### Task Contract`; enforcement is template-only (`/new-plan` scaffold). A criteria-less plan is a process failure (draft them first), not a license to invent a bar.
- **Multi-git-provider support** — projects span GitHub, Bitbucket Cloud, Azure DevOps. Provider is inferred from each project's `repo` remote host (no `git_provider` field). PR/repo tooling routes by provider: `gh` / `bkt` skill / `az repos`. The no-merge guardrail extends to all three (Azure merge = `az repos pr update --status completed`).
- **Skills owned by memory, symlinked into harnesses** — authored skills live in `skills/<name>/` and remote skills materialize in `.skill-cache/<name>/`; each runtime's skills dir holds per-skill symlinks back into those stores. Sync via `scripts/link-skills.sh`.
- **Skill subsystem — validation + on-request self-rating; no write boundary.** Skills carry no write-boundary declaration: `metadata.tier` and the detective git-diff boundary check (the two in-session hooks + `skill-boundary-check.sh`) were removed — the mechanism was detective-only, overlapped execpolicy and the Validator, and didn't fit remote skills (external repos can't carry the field). The floors that remain: **execpolicy** is the destructive-class floor, and the **Validator** catches a skill that mutated something it shouldn't (against the plan's success criteria). `validate-skills.sh` statically checks the schema (name/description, size, placeholders); `new-skill.sh` / `install-skill.sh` author / intake to it. First-party workflow skills carry an **on-request** self-rating block (skill friction, a separate axis from the Validator) — loop membership is **marker-derived**, not a static list; this stays intact. Authoring rule: every script action keeps a hand-editable equivalent (Two-Path principle). Historical design record in the `skill-subsystem` plan (boundary apparatus removed by the `remove-metadata-tier` plan).
- **Skill source model** — skills now have one meaningful axis: **source**. Authored skills live in `skills/`, which is per-instance and gitignored except `.gitkeep`; the old generic/local authored split is retired, so there is no tracked in-tree authored skill store. Remote skills are declared in the per-instance root `skills.toml`, seeded from the tracked `skills.toml.example` catalog on install, fetched into the gitignored `.skill-cache/`, and pinned by a lockfile: *referenced, not forked*. Sharing a skill means publishing it to a remote repo and adding it to the catalog/manifest. Stateful skills persist local data in a gitignored `.skill-data/<name>/` root (`AI_MEMORY_SKILL_DATA`), decoupled from the ephemeral skill dir so remote-referenced skills don't lose state on re-resolve. Enumeration is centralized in `_lib.sh:list_skill_dirs` over the two roots (`skills/` + `.skill-cache/`; override via `AI_MEMORY_SKILL_ROOTS`). `install-skill --remote --save` writes the manifest, `resolve-skills.sh` (+ `sync-system.sh --update`) materialize it, `list-skills` shows provenance. Full reference in `docs/harnesses/claude.md` → Skill Source.
- **Doc-vs-code gate — the `docs/scripts.md` env-var table is machine-checked** (shipped 2026-07-10, the `doc-vs-code-consistency-test` plan). `scripts/check-docs.sh` runs as a `== doc-vs-code ==` stage in `run-tests.sh` and gates the exit code (0 clean / 1 findings / **2 setup error**, so an unparseable table can never read as clean). Two axes over the structured `Var | Default | Used by` table: **forward** (a documented var exists somewhere in the code roots) and **strict** (it appears in the script `Used by` names, **or in any file that script sources, transitively** — "Used by" means *whose behaviour the var affects*, not which file holds the string; `lint-memory.sh` reaches `AI_MEMORY_PROJECTS_ROOT` only via `projects_root()` in `_lib.sh`, and `inject_memory.sh → memory_common.sh → _lib.sh` is a depth-2 chain with a conditional inner source). A `Used by` cell naming no script **fails** unless exempted in `.docscheck-exempt` *with a reason*. Data is fixed rather than the parser (the `TASK / _EXPLORE` shorthand row became two rows). **Scope is narrow by design and stated as a non-goal, not an implied claim:** it catches symbol drift only — never a semantic prose promise (`--dry-run` "mutates nothing" while running `git fetch --tags`), never a hand-written count (delete the count; a count is drift-by-construction), never a var documented outside the table, and never a contradiction with a remote skill in another repo. The motivating bug was `MEMORY_SESSIONS_DIR`: documented in two files, present in **zero** code files, and describing a *removed mechanism* rather than a renamed var.
- **A control is not trusted until it has been watched to fail.** Building the gate above produced three fail-open bugs **inside the control itself**, none found by review, all by injecting a known defect and demanding failure: (1) `check-docs.sh` lives in `scripts/` and its comments cite `MEMORY_SESSIONS_DIR`, so the forward axis **passed for the very var it exists to catch** — the checker certified a ghost because it described it (fixed with `--exclude=check-docs.sh`); (2) a `sed` delimiter collision (`s|…(\.|source)…|`) killed source-following, and a `2>/dev/null` turned the crash into an empty result, producing four false positives that looked exactly like real drift; (3) the test's own "source cycle terminates" assertion was **vacuous** — it passed with or without `closure()`'s visited-set guard, because the runaway recursion nests in `| while` subshells, `fork()` fails, and the right answer emerges from a broken program. Fixed by making the guard observable (`CLOSURE_MAX` → fail-closed exit 2, never report clean from a traversal that didn't terminate). **Generalises: mutation-test every assertion — reintroduce its bug and watch it go red — and treat any component that degrades silently (`ls` under `rtk`, a swallowed stderr, a masked fork failure) as the primary suspect.** Same family as the green-suite and passing-count traps below.
- **Knowledge lifecycle** — per-project `working.md` → `domain/<topic>.md` or project `memory.md` Decisions Log (via `/promote-memory`) → packaged skill (manual graduation when a domain file grows past ~50 lines or holds a coherent reusable pattern).
- **Derived state snapshot (`/state`)** — `scripts/regenerate-state.sh` projects a single on-demand "In Flight" table across all projects (`project | last touched | current goal | open todos`), every column derived (last-touched = newest file mtime — *not* `git log`, since most project trees are gitignored; current goal from `## Current Goal`; open-todo count from `todo.md`). A projection like `index.md` (can't drift), but **on-demand only — never auto-injected**: it surfaces *that* sibling work exists without loading sibling `memory.md` (delegate-don't-load). Output `state.md` is gitignored. Exposed via `/state`.
- **Project categories + activity report** — projects carry an optional `category:` (client/group) in `memory.md` frontmatter. **Per-instance personal data:** the engine supports the field, but real project `memory.md` is gitignored so values never enter git history (the tracked `ai-memory` meta-project stays category-less; `_template` ships a commented placeholder). Set via `/pin --category`, `/new-project`, or hand-edit. Powers category-grouped `/state` (uncategorized last) and **`/activity`** (`regenerate-activity.sh`): the plans *created* in a window (default 30d), grouped by category, scanning live **and** archived plans — the billable/reviewable unit is a **plan** (`created:`), so the report is decoupled from the task backend. Output `activity.md` gitignored, on-demand. One flat category per project.
- **Durable memory names artifacts; it does not store their paths** (decided 2026-07-09) — a path moves when a plan or wiki is archived, so a path written into `memory.md` is a latent lie with a timer on it. Reference the **`versioned-release-channel` plan**, not `archive/plans/versioned-release-channel.md`. The reader (human or agent) resolves a name by looking in the live dir, then `archive/` — **completed work lives in `archive/`, and that is the convention, not something each reference restates.** A task's backend record likewise carries only *intent*; the additional information lives in an **investigation in the tree**, named. Only *machine-consumed* fields carry paths — plan frontmatter (`task_ref`), `todo.md`'s link to its plan — and those point at live files by definition, so they can't rot into the archive. This is why `memory.md` still names `archive/` as a **directory policy** (what it is, what's never read) while never naming a file inside it. Caught by this rule going wrong: `memory.md` simultaneously claimed the release-packaging design record was at `brainstorms/…` (line 54) and `archive/wikis/…` (line 91).
- **Wikis document existing system components, not plan references** — `wikis/` is for durable docs of what the system *is*; comparison/triage/adoption-backlog docs are plan inputs and get archived to `archive/wikis/` once the plan using them ships (the POS triage docs were archived when state-snapshot landed).
- **This meta-project is tracked in-repo** — unlike every other project (external repo + client data → gitignored), `projects/ai-memory` documents *this* repo and carries no secrets, so its durable artifacts are version-controlled via a `.gitignore` carve-out: **tracked** = `memory.md`, `plans/`, `investigations/`, `todo.md`, **and `archive/`** (past work stays visible in git — the memory system simply never *loads* `archive/`, a runtime convention, not a tracking one); **ignored** = only `working.md` (per-machine live scratch). It stays tracked **through open-sourcing** — the plans and investigations *are* the reasoning worth publishing, and the meta-project is the system dogfooding itself. Consequence: routine `memory.md`/`todo.md` edits and plan-archival become git changes — **commit these directly to `main` (no PR)**; reserve a branch + PR for new features / substantive code changes. Direct pushes to `main` remain correct under the release channel: consumers sync to *tags*, never `main`, and `release.sh`'s test-suite guard is the real gate. (Branch protection is currently *impossible* anyway — `release.sh` pushes `main` as part of the cut; the fragments + Action design is what would unlock it.)
- **Open-sourcing: remove content going forward, never scrub** (decided 2026-07-08). The repo is headed public. Existing history and `v1.0.0` keep everything already committed — that history *is* the thought process, and it is deliberately published. Only `identity.md` and *live* client-named references leave the tracked tree going forward; the client-named files under `archive/` stay untouched as audit trail. **`identity.md` was always the wrong thing to track** (see the sync-brick gotcha below), so this is a bug fix that happens to also be an open-source prerequisite. The old claim "repo is private, so internal repo/org names are acceptable to track" is **retired** — it was safe when written and stopped being safe the moment tags became the distribution mechanism.
- **Per-environment config loader; instance config never in tracked memory** — `_lib.sh` and `taskctl` source a gitignored `config.local.sh` next to the tree; ship `config.local.sh.example`. Because the same core/memory is shared across multiple instances, **machine/instance-specific choices live in `config.local.sh`, not in this tracked `memory.md`** — projects root (`AI_MEMORY_PROJECTS_ROOT`), executor (`AI_MEMORY_EXECUTOR`), task provider (`MEMORY_TASK_PROVIDER`), etc. Scripts read them by sourcing `_lib.sh`; the orchestrator reads runtime-derived ones via their tool (e.g. `executor.sh --which`). The committed default root is `$HOME/Projects`.
- **Harness-agnostic, manifest-driven engine** — no harness is privileged. Each lives under `harnesses/<name>/` and is registered by a declarative `manifest` (data, never sourced) with two capability faces: **deliver** (install surfaces — `archetype` `hook`|`file`, `format` `xml`|`md`, `context_target`/`skills_dir`/`commands`…) and **execute** (`exec_*` — the executor contract). `install.sh` is the generic, agent-runnable engine (`--harness <name>` | auto-detect | `--list`): resolve → validate manifest → archetype driver (`scripts/drivers/{hook,file}.sh`) → deliver commands/skills/agents → optional per-harness override. Content selection lives once in `scripts/content-core.sh` (+ `formatters/{xml,md}.sh`); the file-archetype md context builds once via `scripts/build-context-md.sh` (Codex's `codex-mem.sh` calls it). **Registered today:** `claude` (hook/xml), `codex` (file/md), `antigravity` (hook/xml — live injection + enforced executor; see the Antigravity decision below). Adding a harness is a manifest, not engine code. Full design + phase record in the `make-memory-engine-harness-agnostic` plan.
- **Codex adapter** — `codex-mem.sh` (a `file`-archetype launch wrapper) regenerates `~/.codex/AGENTS.md` on every launch via the shared `build-context-md.sh` (no hooks in Codex; only the injection mechanism differs). `--executor` = `exec --sandbox workspace-write` network-on; `--executor-bare` strips the memory merge (~13k tokens) for read-only review subagents. Mid-session memory edits don't appear until relaunch. Codex is now a fully manifest-registered harness (install target + executor), not a bespoke bolt-on.
- **Antigravity harness — hook archetype, live injection + enforced executor** — Antigravity (`agy`) is a genuine `hook` archetype, not a materialized file: memory injects **live per model call** via a `PreInvocation` hook in `~/.gemini/config/hooks.json` (built from the shared `content-core.sh` + `xml.sh` — the same `<memory:*>` payload as Claude; `invocationNum==0` → full, else breadcrumb). Its hook payload carries **no workspace handle**, so `agy.sh` resolves the active project at launch and exports it (`AI_MEMORY_PROJECT`/`MEMORY_DIR`) for the hook to read; the static workflow-rules base is a hand-owned `~/.gemini/config/AGENTS.md` (agy has no `AGENTS.local.md`). It is also the **2nd enforced executor after codex**: a `PreToolUse` guard (`AI_MEMORY_ROLE`-gated, so interactive `agy` stays unguarded) enforces the shared `scripts/deny-list.txt` for both roles and a read-only **allowlist** for `explore` — giving Antigravity a real `exec_readonly` instead of degrading to Claude Explore. The deny-list is now a **shared shipped artifact** (seeding a future manifest `guard` capability). It also ships a **memory-aware statusline** (`statusline.sh`, registered via `settings.json → statusLine` by an idempotent install merge that preserves existing keys) surfacing the active project + folder + branch/model/context/agent-state; emoji glyphs by default (matching Claude — `USE_NERD_FONTS=true` opts into Nerd Font). Full design + phase record in the `antigravity-hook-archetype` and `antigravity-statusline` plans.
- **Hook layer — declarative, manifest-driven, shared where isomorphic** — hook wiring is a capability, not per-harness code. Canonical roles (`session_bootstrap`, `per_turn_inject`, `infra_guard`, `task_tool_block`, `compaction_recovery`) → a manifest `[hooks]` role→`event[:matcher]` map (`manifest_hooks`) → `drivers/hook.sh` writes each harness's native hooks-file schema with **no hardcoded event names**. The **behavioral** scripts are shared across the **isomorphic Claude+Codex pair** — `scripts/hooks/{inject,guard,lib}.sh`, with format chosen at the `content-core`/formatter boundary (`AI_MEMORY_HOOK_FORMAT` xml|md, `AI_MEMORY_HOOK_EVENT`); `inject.sh` is byte-identical to Claude's former `inject_memory.sh`, so the migration was behaviour-preserving. **Registration schemas differ per harness, so per-harness writer branches are accepted, not a failure** — Claude's `settings.json` and Codex's `hooks.json` share one schema (`_hook_register_native_json`: fail-closed, idempotent, preserves sibling keys + unrelated user hooks), while Antigravity keeps its own `_hook_register_json` + `PreInvocation`/XML path. **No canonical adapter is built** (the descope: share the isomorphic pair, special-case the one divergent harness — a fixed ≤5-harness set doesn't earn the abstraction; onboarding-the-Nth-harness is a cost rarely paid). Codex is a **file+hook hybrid** — `AGENTS.md` static base + native hooks for the dynamic layer (`per_turn_inject`, `infra_guard`); its hook trust is a one-time `/hooks` step (interactive) + `--dangerously-bypass-hook-trust` (executor), because `requirements.toml` managed-trust is **MDM-only, empirically dead on a personal install**; the hook capability is install-time version-gated (`hooks_min_version`), falling back to file-only below the floor. The `infra_guard` role **subsumes** the former standalone guard-capability task. **Guard stdin/registration schemas are harness-specific and were the main bug surface** — Codex's `PreToolUse` command lives at `tool_input.command` (not Antigravity's `toolCall.args.CommandLine`), verified against real stdin (a wrong path fails OPEN). **Open:** `compaction_recovery` is live on Claude (SessionStart sentinel, consumed by the shared `inject.sh`) but not yet on Codex — arming the sentinel on a Codex compaction is a tracked follow-up. Full design + phase record in the `hook-standardization` plan; Codex hook schema/trust/stdin reference in `domain/codex.md`.
- **Hook layer — share the isomorphic, special-case the divergent; don't abstract over a bounded harness set.** The `[hooks]` role→event map lives in each manifest and `drivers/hook.sh` reads it (no hardcoded event names), but sharing stops at what is genuinely isomorphic. Claude and Codex hooks speak the *same* contract (identical stdin fields + `additionalContext` output), so they share one behavioral script set (`scripts/hooks/{inject,guard}.sh`); Antigravity diverges (`PreInvocation`, no `session_id`, XML) and keeps its **own thin registration/injection path**, sharing only the content layer (`content-core.sh` + formatters) it already shares — **no canonical adapter is built to force it into a shared contract.** The rejected ambition was a fully-general driver + adapter making every harness manifest-only: with a **fixed, author-controlled set of ≤5 harnesses**, a small per-harness registration branch is more readable than a data-driven resolver whose sole exception is a third of the set, and "onboarding the Nth harness is free" optimizes a cost rarely paid. Value concentrates in the shared *content* layer (where drift actually causes bugs) and in Codex *gaining* real hook capabilities (live per-turn inject, mid-session project switch, real infra gate) — each justifies itself independently of standardization. Generalises: an abstraction whose only justification is "cheap to add the Nth case" is over-generalization when N is small and bounded. Design + phase record in the `hook-standardization` plan.
- **Versioned release channel — a git tag IS the release; no zip, no separate engine root** — instances sync to tested cuts, never a moving `main`. `AI_MEMORY_CHANNEL` (per-instance, in `config.local.sh`): `release` (default) checks out the latest **stable** `v*` tag; `dev` keeps the ff-pull. `--to <ref>` is a one-shot pin/dogfood/bisect that never changes the channel, so an instance can't be stranded on a stale ref. Git supplies for free what a zip design must build (version identity = `git describe`, rollback = checkout, integrity = the object store, local-edit detection = a refusing checkout, stale-file deletion = checkout) — so the **zip + `.engine/<ver>/current` design was rejected**: it solved artifact-cleanliness and no-git distribution, problems that don't exist while every consumer is an authed clone. Data that git can't move rides a **migration runner**: `migrations/<semver>-<slug>.sh`, forward-only, idempotent, keyed on migration *files* not tags (so untagged dogfood refs migrate identically), gated by the gitignored `.applied-version` high-water marker written after **each** success. A downgrade therefore moves code back but leaves data migrated — safe **only** because of the **N/N+1 rule** (a migration must not break the previous release's code; ship the compat fallback in N, flip in N+1). That rule lives in `migrations/README.md` + `UPGRADING.md` and **nothing enforces it** — the Validator is the backstop. `release.sh` cuts the tag: guarded (dirty / non-`main` / diverged from origin / tag exists / non-monotonic version / failing suite), **idempotent and resumable** (already-released ⇒ no-op; tag-without-push ⇒ resume at push, skipping the suite; release-commit-without-tag ⇒ resume at tag, *running* the suite), and it **refuses outright when `AI_MEMORY_ROLE` is set** — so an executor structurally cannot publish, and a future GitHub Action (not an executor) still can. Design record: the `versioned-release-packaging` wiki; the deferred external-user zip thread is its §7.
- **Non-goals:** no `PreToolUse` deny enforcing `todo.md` (documented convention only). ~~No bootstrap script.~~ **Reversed (Phase 5, harness-agnostic work):** `install.sh` IS the agent-runnable, manifest-driven bootstrap (`--harness <name>` | auto-detect | `--list`) — an agent inside any harness runs it to wire that harness up; README demoted to backup.

## Known Constraints / Gotchas
- **This repo IS under git, but personal content is `.gitignore`d.** Authoritative list is `.gitignore` — verify there. Tracked (distributable source): `scripts/`, `harnesses/`, `agents/`, `install.sh`, `README.md`, `identity.template.md`, `config.local.sh.example`, `skills.toml.example`, `projects/_template`, `domain/_template.md`, and `skills/.gitkeep`. Ignored (personal/runtime): `.claude/`, `.sessions/`, `/identity.md` (per-instance since v1.0.0 — see the sync-brick gotcha), `/index.md`, `/domain/*` (except `_template.md`), `/projects/*` (except `_template` **and `ai-memory`**), `/tasks/`, `/archive/`, `/config.local.sh`, `/skills/*` (authored per-instance skills; `.gitkeep` tracked), `/.skill-cache/` (materialized remote skills), `/docs/superpowers/`. Live per-engagement memory never gets committed **except this `ai-memory` meta-project**. Never commit secrets.
- **Scripts target macOS `bash` 3.2** — no `mapfile`, no associative arrays. Resolve the tree via `MEMORY_DIR`. `_lib.sh` self-locates `MEMORY_DIR` via `BASH_SOURCE` (correct under bash, empty under the Bash-tool's interactive zsh — don't sanity-check `. scripts/_lib.sh` from zsh).
- **`archive/` is never read unless the user explicitly asks** — and never deleted during a "reorganize memory" pass (audit trail).
- **Frontmatter is the contract** — missing/malformed fields break the index; `lint-memory.sh` catches it. `_template/` is excluded from index/lint/regeneration.
- **Working memory is per-project; its content rides the full payload injected once per session (and on `@memory`/post-compaction), NOT every prompt.** Ordinary prompts carry only the `<memory:active>` breadcrumb, which advertises the working *path* (the checkpoint write-target), not its content. (Same across harnesses: Claude/Antigravity inject full on first prompt/invocation then breadcrumb; Codex bakes it into `AGENTS.md` once at launch.) Keep it short anyway — it sits in the once-per-session cached prefix and re-ships whole on every `@memory`/compaction recovery, so bloat inflates the session-start payload and each recovery (not every request). Two terminals on the same checkout both writing `working.md`: last write wins (no merge; acceptable for solo use — separate git worktrees get their own `working.<key>.md` overlay).
- Slash commands are indexed at session start — a new command needs a session restart to autocomplete.
- Oversized hook stdout is routed to a `<persisted-output>` file with a 2KB preview (the hard 10k cap is gone), but a bloated `working.md` still re-ships whole in the full payload (session start, `@memory`, post-compaction).
- **For a security control, a green test suite is where validation STARTS, not ends.** The executor deny-list matcher (`scripts/deny-match.sh`, PR #51) took **7 adversarial validation rounds** to harden; each round found bypasses **after the suite was already green**, 24 classes total, and **3 were introduced by the fix for an earlier finding**. Two lessons generalise beyond this file: (1) **a fix that NARROWS a matcher to kill a false positive is the likeliest place to create a false negative** — both directions are invisible without an adversary actively trying to break it, so pair every FP fix with an FN probe and vice-versa; (2) **cross-*family* review beats cross-model** — rounds 6-7 ran on Fable (vs the Opus/Sonnet rounds 1-5) and caught a blind spot shared across the entire Opus family, including a false positive baked into the fix *sketch* itself (`timeout 5 eval "…"` — a wrapper can't exec the `eval` builtin). This is the empirical case behind the Validator role defaulting to a different plane than the executor. Method rule reinforced: **verify shell/tool semantics by PROBING (`echo PWNED`), never by reasoning** — `eval` flattens quoting, `find -exec` preserves it (opposite, one line apart); two "test failures" one round were wrong expectations, not bugs. The matcher is a **backstop against an honest agent, not a sandbox** — base64 / script-file / `ssh host …` / `eval "$(…)"` obfuscation is deliberately out of scope (codex's execpolicy is the destructive-class floor).
- **A test file the runner's glob doesn't reach is silently ungated — the suite reports green and proves nothing.** `run-tests.sh` globbed only `scripts/tests/test_*.sh`, so `scripts/taskprovider/tests/` (4 Python files) had **never executed**; the suite printed `32 passed` before *and after* a whole new test file was added to it. Fixed 2026-07-09 (#49): a hermetic `== taskprovider (python) ==` stage that gates the exit code, plus `scripts/check-provider-tests.sh` enforcing `providers/<name>.py` ↔ `tests/test_<name>.py` (adding a provider needs no factory edit *by design*, so nothing else would notice one landing untested). **The trap is not closed, only narrowed:** any future test living outside `scripts/tests/*.sh` or `scripts/taskprovider/tests/` is still invisible to the runner. Same class as the shellcheck / doc-vs-code prevention tasks — a control everyone believes is enforced, silently isn't. A test asserting a *count* would not catch it either; only asserting that a named file **ran** would.
- **A recorded measurement is where planning STARTS, not ends — re-measure before you act on it, especially when it is the fact that justifies the work.** The `shellcheck-gate` brainstorm wrote its baseline into working memory, annotated it *"don't re-measure"*, and was wrong on the one claim the task rested on: that `SC2015 × 9` was the `A && B`-as-last-statement bug that aborted `install.sh`, so a gate "would pay for itself on exactly its motivating bug." In fact SC2015 is a **different** check (`A && B || C` is not if-then-else), it is severity **`info`, not `warning`** (so the proposed `-S warning` gate catches **zero** of them), and all nine sites are in test files, not `install.sh` or a driver. The real bug — `drivers/hook.sh`, a guard-registration notice as a function's last statement returning 1 under `set -euo pipefail` — is invisible to shellcheck at default severity *and* under `-o all` (including `check-set-e-suppressed`), verified by probe. It had been live and unnoticed since the hook archetype shipped; found and fixed in `v1.2.0` only because the baseline was re-measured. **Two generalisations.** (1) This is the deny-list lesson one level up: there, a green suite felt like proof the matcher was safe; here, a written-down number felt like proof the task was justified. Both are artefacts that *record* a verification and then get trusted in place of one — and the do-not-recheck annotation is the tell, not the safeguard. (2) **A measurement taken to justify a task is the one most likely to be motivated** — it was collected by someone who already wanted the task to be worth doing, so re-derive it before the plan hardens around it. Corollary for planning: state which control catches which bug **class**, and check that pairing empirically; a linter, a type checker and a test are not interchangeable, and picking the wrong one ships a gate that proves nothing.
- **Codex execpolicy** `decision` keyword is `"forbidden"` (not `deny`/`reject`/`block`); invalid values error at `codex exec` time, not parse time. Executor invocation needs `--skip-git-repo-check` when the workdir isn't a git repo.
- **Codex `exec` (headless) fires `SessionStart(source=startup)` AND injects its `additionalContext` into model context — verified by live probe (2026-07-16, codex 0.144.4; floor 0.135.0).** Probe: isolated `CODEX_HOME` with **no `AGENTS.md`**, a `SessionStart` hook that emitted a sentinel-only string (`ZANZIBAR7`); a one-shot `codex exec` echoed the sentinel back — so the *only* channel it could have arrived by was the hook, and headless runs get hook-injected context, not just interactive ones (codex logs `hook: SessionStart Completed`). This is the capability that unblocks moving the `AGENTS.md` base load off the launch wrapper onto a `SessionStart` hook for **both** faces — mirroring Antigravity's hand-owned-static-base + live-inject model (retiring `codex-mem.sh`'s memory-tree build; the wrapper stays only for exec sandbox/network flags). Design in the `codex-sessionstart-base-load` plan. **Still unproven there:** whether SessionStart `additionalContext` survives a *compaction* on codex — on Claude it does **not** (that's why the compact-branch sentinel→`UserPromptSubmit` dance exists), and only startup is probe-confirmed on codex; the compact path must be re-probed before the sentinel dance can be simplified.
- **Notion provider:** the API can add/reorder but cannot rename/delete select/status options (rename = add + reorder + leave old vestigial). For `/task`/`/start` to route to Notion, creds + `MEMORY_TASK_PROVIDER=notion` + `NOTION_STATUS_KIND` must be in `~/.zshenv` (NOT `.zshrc`) — the Bash tool's non-interactive zsh sources `.zshenv`.
- **Notion refs are ALWAYS the full page UUID, never the 8-char prefix.** A short prefix is *ambiguous* (e.g. `38ef6850` matches both the harness-agnostic task and the delete/trash task) **and** the API rejects it (400 `should be a valid uuid`). `notion.py` guards `get`/`update`/`set_status` (`_require_full_ref`) and fails early with an actionable message; record the full ref (from `taskctl list`/`get`) in `memory.md` and plan `task_ref` frontmatter. The local provider uses slug refs — this guard is Notion-only.
- **Plan-mode plan path:** the harness default (`~/.claude/plans/<slug>.md`) conflicts with the canonical path (`projects/<active>/plans/<name>.md`). In plan mode, always write to the memory-system path.
- **`config.local.sh` beats the environment.** `_lib.sh:10` sources it *after* the process env exists, so an `export`ed value there overrides an env var of the same name. `AI_MEMORY_CHANNEL=release sync-system.sh` is **silently ignored** on any instance whose config sets the channel — not an error, ignored. To change an instance's channel, edit `config.local.sh`. (This checkout is pinned `dev` so the `release` default can never flip the working tree onto a tag.)
- **`git tag -a -m` strips every line starting with `#`** (default `--cleanup=strip` treats them as comments). A changelog body's markdown headings vanish from the annotated tag while the bullets survive, so the tag *looks* fine — and `git tag -l -n` reformats the message, hiding the loss; only `git cat-file tag` shows the raw object. `release.sh` uses `--cleanup=verbatim -F -` on **both** the normal and resume-at-tag call sites. This shipped in `v1.0.0` (retagged) because 107 assertions checked that a tag was created and never checked what it *said*.
- **`--to <branch>` checks out that ref as-is and does not fast-forward it.** On a consumer whose local `main` is stale, `--to main` dogfoods yesterday's code. Use `--to origin/main`.
- **`v1.0.0` is a trap tag — never point an instance at it.** It predates the `identity.md` untracking, so checking it out **silently overwrites** a personalised `identity.md` with the tag's tracked copy: no conflict, no warning, clean tree afterwards. Generalises: **a downgrade can clobber any file the older tag tracked and the newer one does not**, because `git checkout` writes tracked files over ignored ones without complaint. `v1.1.0` is the first tag safe to sync a consumer to. Documented in `UPGRADING.md` and the `/sync-system` command.
- **Docs rot within hours when phases ship serially.** Phase 4 (#43) documented a world Phase 5 changed an hour later — `UPGRADING.md` shipped asserting "there are currently no stable `v*` tags". And the `/sync-system` slash command still described the pre-Phase-1 script two days on, because **nothing tests a command doc against the script it invokes**. When a phase changes runtime behavior, re-read the *command* docs, not just the reference docs.
- **A tracked `identity.md` bricks the release channel.** `install.sh:219` tells the user "Then: edit identity.md", `identity.md` is tracked, and `sync-system.sh:dirty_tracked_guard` aborts on any modified *tracked* file — so an instance can never sync once personalised. Reproduced in a fixture. Fix (locked, task `397f6850-c619-81dd-a098-f9a96429af77`): untrack it; `install.sh` already seeds it from `identity.template.md` when missing. **Untracking deletes it from every tree on the next checkout that crosses the commit** — `git pull`, `git reset --hard`, a branch switch, not just `sync-system.sh` — because the file is still tracked in the *old* `HEAD`. It bit this dev checkout minutes after the fix merged (recovered byte-identical from `git show v1.0.0:identity.md`, since the blob lives on in the history we're publishing). A migration can't rescue it: the runner executes *after* checkout, when the file is already gone. That needs an `UPGRADING.md` note, not a `migrations/*.sh`. **Generalises: any tracked file the user is told to edit is a latent sync-brick; any tracked-file *removal* is data loss the migration runner structurally cannot cover; and the loss is silent for anyone whose copy was unmodified — `install.sh` reseeds the template over it without warning.**
- **A git tag ships a whole tree, and there is no build step to filter it.** `git archive` honours `.gitattributes export-ignore`; `git clone`/`git checkout <tag>` do not (and there is no `.gitattributes` here). So under tag-based distribution **"tracked" means "published in every release, immutably"** — the zip design's leak-guard has no equivalent, because tags have no build step to run one in. Consequence: the tracking decision *is* the distribution decision, and deferring it costs one more frozen copy per tag.
- **A GitHub force-push does not remove anything.** `refs/pull/*` (44 of them here) are GitHub-controlled, survive `filter-repo` + force-push, stay fetchable, and their "Files changed" pages keep rendering old diffs. Verified: PR #40's head still carries 62 `projects/ai-memory/` files. Only deleting and recreating the remote drops them — which also destroys every PR discussion. Relevant to any future scrub; **moot for the current plan, which removes content going forward only.**
