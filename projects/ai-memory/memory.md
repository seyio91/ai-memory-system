## What It Is
The markdown-only memory system for Claude Code â€” no vector DB, no daemons, no MCP server, just files + hooks + macOS-`bash`-3.2 shell scripts. This is a **meta-project**: the project being tracked *is* the memory tooling. Single owner (Seyi). Authoritative spec is the repo `README.md` (a lean landing page) plus the deep-reference pages under `docs/` â€” `docs/harnesses/<name>.md` per harness, mirroring the multi-harness direction.

Three layers (Karpathy LLM-Wiki pattern):
- **Schema** â€” `identity.md` (hard rules, injected once/session), `~/.claude/CLAUDE.md`. Outranks everything.
- **Wiki** â€” `domain/*.md` (cross-project) + `projects/*/memory.md` (engagement-specific). Durable.
- **Scratchpad** â€” `projects/*/working.md`, injected every prompt while non-empty; matures into the wiki via `/promote-memory`.

Moving parts: one `UserPromptSubmit` hook (`~/.claude/hooks/inject_memory.sh`) emits `<memory:*>` blocks; slash commands in `~/.claude/commands/`; scripts in `scripts/` (`_lib.sh`, `regenerate-index.sh`, `lint-memory.sh`, `archive-cleanup.sh`, `new-project.sh`, `memory-pin.sh`, `link-skills.sh`, `executor.sh`, `codex-mem.sh`, the `taskprovider/` package + `taskctl`) with dependency-free bash-3.2 tests under `scripts/tests/`.

## Architecture Decisions
- **Markdown over a DB** â€” every editor/diff/grep works; zero infra; git-trackable.
- **Hook-injected, not retrieved** â€” context arrives in-band. Full payload (identity â†’ project memory â†’ index â†’ working) is injected at SessionStart, on `@memory`, and on the first prompt after compaction; every other prompt gets only the lightweight `<memory:active>` breadcrumb. Domain files are the exception: lazy-read on index trigger-match, never auto-injected.
- **Frontmatter-driven catalog** â€” `index.md` AUTOGEN block is regenerated from frontmatter (`/reindex`), so it can't lag the files. **Exactly one canonical fence:** `<!-- BEGIN AUTOGEN -->` / `<!-- END AUTOGEN -->` (a `## Projects` roster + a `## Domain` table). The older single combined-table `<!-- AUTOGEN:START -->` / `<!-- AUTOGEN:END -->` block is **retired** â€” `regenerate-index.sh` never touches it, so any surviving copy silently rots and duplicates the live tables; instances carrying it must delete it. Both scaffold files are excluded from the catalog: the projects loop skips `projects/_template/`, and the **domain loop skips `domain/_template.md`** (its literal `topic: <topic>` frontmatter otherwise leaks a placeholder row) â€” mirror both guards in any regenerator change, and guard the domain exclusion with a test.
- **Lean index** â€” `index.md` is a path-less roster; paths are derivable (`projects/<name>/memory.md`, `domain/<topic>.md`), metadata (`tags`/`repo`/`repo_path`, domain `triggers`) stays in the source files. Codex's `AGENTS.md` keeps absolute paths (its shell tool reads by path).
- **Per-session injection markers** (`~/.claude/memory_sessions/<session_id>`, self-expire after 2 days) â€” once-per-session blocks don't re-inject; concurrent sessions don't collide.
- **Compaction recovery** â€” `PreCompact` stdout can't inject; the reliable channel is `UserPromptSubmit`. `SessionStart source=compact` writes a per-session sentinel; the next prompt re-injects the full payload once and clears it.
- **Project detection per-prompt** â€” the hook reads `cwd`, walks up for a harness-neutral `.agents/memory-project` marker (legacy `.claude/memory-project` still read as a fallback at the same level); no marker â†’ no project (memory dormant, no global fallback, so concurrent repos never collide). `.active_project` machinery still exists but is left empty by deliberate decision.
- **Bidirectional repoâ†”project map** â€” forward marker `.agents/memory-project` (harness-neutral; `memory-pin.sh` migrates a legacy `.claude` one on pin; `scripts/migrate-marker.sh` bulk-migrates) + reverse `repo`/`repo_path` frontmatter, both written by `/pin` (`memory-pin.sh`). `resolve_repo_path` resolves a checkout per-environment: `repo_path` relative to `projects_root()` (`AI_MEMORY_PROJECTS_ROOT`, default `$HOME/Projects`), git-remote `repo` as portable fallback. `lint-memory.sh` catches reverse-map drift. **`repo_path` is stored relative, never absolute** â€” the env-var root supplies the prefix so paths resolve identically across host and sandbox. **Self-referential exception:** the `ai-memory` meta-project's checkout *is* the memory tree, so its `repo_path` is the sentinel `$MEMORY_DIR` (and `$MEMORY_DIR/<sub>` is supported) â€” `resolve_repo_path` and `lint-memory.sh` resolve it against `MEMORY_DIR` rather than `projects_root()`, keeping the committed value machine-independent (each machine sets `MEMORY_DIR` to wherever its tree lives).
- **Distributed cross-project relationships** â€” a relationship lives in the project where the work *starts* (`## Related Projects`), never in an umbrella. Siblings are **delegated to subagents, never loaded** into the orchestrator thread; plan-set execution pauses at human/CI gates.
- **Orchestrator / Executor / Validator workflow** â€” Claude main session = orchestrator; an executor runs delegated work; the validator resolves through the **same configured executor** (re-run `executor.sh --which`) as a **separate, fresh invocation** â€” independence comes from the extra invocation, not a different tool â€” checking output against the plan's `## Success criteria`. Harness `TaskCreate`/`TaskUpdate` are banned (enforced by `block_task_tools.sh`); `todo.md` is the single source of truth. Archive is never read unless asked. Executors are blocked from any apply/merge/destructive action on running infrastructure.
- **Executor: two roles, manifest-resolved.** `scripts/executor.sh` exposes `--role task` (write-capable) and `--role explore` (read-only), each configured as a `harness[:model]` value (`AI_MEMORY_EXECUTOR_TASK` / `AI_MEMORY_EXECUTOR_EXPLORE`, falling back to the legacy single `AI_MEMORY_EXECUTOR`, then `claude-subagent`). A harness resolves **through its `harnesses/<name>/manifest` `exec_*` block** â€” no per-harness code (the old codex special-case is gone): `exec=subagent` â†’ subagent plane; else `exec_cmd` (task) / `exec_readonly` (explore), gated on `exec_probe`. A harness with no read-only mode is **skipped for `explore`** (degrades to the Claude Explore agent), never run write-capable. Model threads via `exec_model_flag`; the subagent plane surfaces `subagent:<model>`. `--which` prints `subagent[:model]` | `cli:<name>`; `--run` execs or bounces to the subagent (exit 3). **Which executor a given instance uses is config-driven, never recorded here** (set the vars in gitignored `config.local.sh`; the orchestrator discovers via `executor.sh --role <r> --which`). The codex deny-rules file is optional codex-only hardening; the no-merge deny-list is restated in every delegation prompt regardless of executor.
- **Task-provider layer + `/task`/`/start`** â€” pluggable capture/track/execute subsystem; stdlib-only Python `scripts/taskprovider/` (generic registry: env value `MEMORY_TASK_PROVIDER` *is* the provider module â†’ no factory edit to add a backend), `providers/local.py` (flat `tasks/<slug>.md`) + `providers/notion.py` (urllib-only). Backend is a projection + intake queue, push-dominant, never bidirectional. `scripts/taskctl` bash wrapper; `/task` (add|list|done|archive|show), `/start <ref>` (runs the brainstorm gate â†’ scaffolds a plan into the task's own project).
- **Brainstorming skill (gated Tier-3 design pass)** â€” runs the collaborative design pass only for Tier-3 feature tasks with open design questions; silent on Tier 1/2 and settled Tier-3. Terminal handoff is `/new-plan`, folding the approved design into the plan's Goal/Success criteria/Design/Risks. Orchestrator-only, no separate committed spec.
- **Task Contract â€” success criteria** â€” plan-tier tasks carry explicit checkable success criteria (the Validator's contract). Policy in `identity.md` â†’ `### Task Contract`; enforcement is template-only (`/new-plan` scaffold). A criteria-less plan is a process failure (draft them first), not a license to invent a bar.
- **Multi-git-provider support** â€” projects span GitHub, Bitbucket Cloud, Azure DevOps. Provider is inferred from each project's `repo` remote host (no `git_provider` field). PR/repo tooling routes by provider: `gh` / `bkt` skill / `az repos`. The no-merge guardrail extends to all three (Azure merge = `az repos pr update --status completed`).
- **Skills owned by memory, symlinked into harnesses** â€” canonical store `skills/<name>/` under the memory root; each runtime's skills dir holds per-skill symlinks back into it (one source of truth, harness-agnostic). Sync via `scripts/link-skills.sh`.
- **Skill subsystem â€” declared, enforced write boundary + self-rating.** Every skill carries `metadata.tier` (`target-read-only` | `target-write`), a neutral label enforced harness-agnostically â€” *not* Claude's `allowed-tools`. Three write zones: the target tree (tier-gated) Â· the skill's own `skills/<name>/` folder (always writable, no declaration) Â· everything else off-limits. Enforcement is layered and *detective*: execpolicy (destructive floor) + a post-run git-diff boundary check (in-session PostToolUse/Stop hooks; Codex executor trigger deferred) + the Validator (correctness). `validate-skills.sh` statically checks the schema; `new-skill.sh` / `install-skill.sh` author / intake to it. First-party workflow skills also carry an **on-request** self-rating block (skill friction, a separate axis from the Validator); loop membership is **marker-derived**, not a static list. Authoring rule: every script action keeps a hand-editable equivalent (Two-Path principle). Shipped in PR #8; full design + decision record in `archive/plans/skill-subsystem.md`.
- **Knowledge lifecycle** â€” per-project `working.md` â†’ `domain/<topic>.md` or project `memory.md` Decisions Log (via `/promote-memory`) â†’ packaged skill (manual graduation when a domain file grows past ~50 lines or holds a coherent reusable pattern).
- **Derived state snapshot (`/state`)** â€” `scripts/regenerate-state.sh` projects a single on-demand "In Flight" table across all projects (`project | last touched | current goal | open todos`), every column derived (last-touched = newest file mtime â€” *not* `git log`, since most project trees are gitignored; current goal from `## Current Goal`; open-todo count from `todo.md`). A projection like `index.md` (can't drift), but **on-demand only â€” never auto-injected**: it surfaces *that* sibling work exists without loading sibling `memory.md` (delegate-don't-load). Output `state.md` is gitignored. Exposed via `/state`.
- **Project categories + activity report** â€” projects carry an optional `category:` (client/group) in `memory.md` frontmatter. **Per-instance personal data:** the engine supports the field, but real project `memory.md` is gitignored so values never enter git history (the tracked `ai-memory` meta-project stays category-less; `_template` ships a commented placeholder). Set via `/pin --category`, `/new-project`, or hand-edit. Powers category-grouped `/state` (uncategorized last) and **`/activity`** (`regenerate-activity.sh`): the plans *created* in a window (default 30d), grouped by category, scanning live **and** archived plans â€” the billable/reviewable unit is a **plan** (`created:`), so the report is decoupled from the task backend. Output `activity.md` gitignored, on-demand. One flat category per project.
- **Wikis document existing system components, not plan references** â€” `wikis/` is for durable docs of what the system *is*; comparison/triage/adoption-backlog docs are plan inputs and get archived to `archive/wikis/` once the plan using them ships (the POS triage docs were archived when state-snapshot landed).
- **This meta-project is tracked in-repo** â€” unlike every other project (external repo + client data â†’ gitignored), `projects/ai-memory` documents *this* repo and carries no secrets, so its durable artifacts are version-controlled via a `.gitignore` carve-out: **tracked** = `memory.md`, `plans/`, `todo.md`, **and `archive/`** (past work stays visible in git â€” the memory system simply never *loads* `archive/`, a runtime convention, not a tracking one); **ignored** = only `working.md` (per-machine live scratch). Repo is private, so internal repo/org names are acceptable to track. Consequence: routine `memory.md`/`todo.md` edits and plan-archival become git changes â€” **commit these directly to `main` (no PR)**; reserve a branch + PR for new features / substantive code changes.
- **Per-environment config loader; instance config never in tracked memory** â€” `_lib.sh` and `taskctl` source a gitignored `config.local.sh` next to the tree; ship `config.local.sh.example`. Because the same core/memory is shared across multiple instances, **machine/instance-specific choices live in `config.local.sh`, not in this tracked `memory.md`** â€” projects root (`AI_MEMORY_PROJECTS_ROOT`), executor (`AI_MEMORY_EXECUTOR`), task provider (`MEMORY_TASK_PROVIDER`), etc. Scripts read them by sourcing `_lib.sh`; the orchestrator reads runtime-derived ones via their tool (e.g. `executor.sh --which`). The committed default root is `$HOME/Projects`.
- **Harness-agnostic, manifest-driven engine** â€” no harness is privileged. Each lives under `harnesses/<name>/` and is registered by a declarative `manifest` (data, never sourced) with two capability faces: **deliver** (install surfaces â€” `archetype` `hook`|`file`, `format` `xml`|`md`, `context_target`/`skills_dir`/`commands`â€¦) and **execute** (`exec_*` â€” the executor contract). `install.sh` is the generic, agent-runnable engine (`--harness <name>` | auto-detect | `--list`): resolve â†’ validate manifest â†’ archetype driver (`scripts/drivers/{hook,file}.sh`) â†’ deliver commands/skills/agents â†’ optional per-harness override. Content selection lives once in `scripts/content-core.sh` (+ `formatters/{xml,md}.sh`); the file-archetype md context builds once via `scripts/build-context-md.sh` (Codex's `codex-mem.sh` calls it). **Registered today:** `claude` (hook/xml), `codex` (file/md), `antigravity` (hook/xml â€” live injection + enforced executor; see the Antigravity decision below). Adding a harness is a manifest, not engine code. Full design + phase record in `archive/plans/make-memory-engine-harness-agnostic.md`.
- **Codex adapter** â€” `codex-mem.sh` (a `file`-archetype launch wrapper) regenerates `~/.codex/AGENTS.md` on every launch via the shared `build-context-md.sh` (no hooks in Codex; only the injection mechanism differs). `--executor` = `exec --sandbox workspace-write` network-on; `--executor-bare` strips the memory merge (~13k tokens) for read-only review subagents. Mid-session memory edits don't appear until relaunch. Codex is now a fully manifest-registered harness (install target + executor), not a bespoke bolt-on.
- **Antigravity harness â€” hook archetype, live injection + enforced executor** â€” Antigravity (`agy`) is a genuine `hook` archetype, not a materialized file: memory injects **live per model call** via a `PreInvocation` hook in `~/.gemini/config/hooks.json` (built from the shared `content-core.sh` + `xml.sh` â€” the same `<memory:*>` payload as Claude; `invocationNum==0` â†’ full, else breadcrumb). Its hook payload carries **no workspace handle**, so `agy.sh` resolves the active project at launch and exports it (`AI_MEMORY_PROJECT`/`MEMORY_DIR`) for the hook to read; the static workflow-rules base is a hand-owned `~/.gemini/config/AGENTS.md` (agy has no `AGENTS.local.md`). It is also the **2nd enforced executor after codex**: a `PreToolUse` guard (`AI_MEMORY_ROLE`-gated, so interactive `agy` stays unguarded) enforces the shared `scripts/deny-list.txt` for both roles and a read-only **allowlist** for `explore` â€” giving Antigravity a real `exec_readonly` instead of degrading to Claude Explore. The deny-list is now a **shared shipped artifact** (seeding a future manifest `guard` capability). Full design + phase record in `archive/plans/antigravity-hook-archetype.md`.
- **Non-goals:** no `PreToolUse` deny enforcing `todo.md` (documented convention only). ~~No bootstrap script.~~ **Reversed (Phase 5, harness-agnostic work):** `install.sh` IS the agent-runnable, manifest-driven bootstrap (`--harness <name>` | auto-detect | `--list`) â€” an agent inside any harness runs it to wire that harness up; README demoted to backup.

## Known Constraints / Gotchas
- **This repo IS under git, but personal content is `.gitignore`d.** Authoritative list is `.gitignore` â€” verify there. Tracked (distributable source): `scripts/`, `harnesses/`, `agents/`, generic `skills/`, `install.sh`, `README.md`, `identity.md`, `config.local.sh.example`, `projects/_template`, `domain/_template.md`. Ignored (personal/runtime): `.claude/`, `.sessions/`, `/index.md`, `/domain/*` (except `_template.md`), `/projects/*` (except `_template` **and `ai-memory`**), `/tasks/`, `/archive/`, `/config.local.sh`, `/skills/fiter-infrastructure-analyzer/`, `/docs/superpowers/`. Live per-engagement memory never gets committed **except this `ai-memory` meta-project**. Never commit secrets.
- **Scripts target macOS `bash` 3.2** â€” no `mapfile`, no associative arrays. Resolve the tree via `MEMORY_DIR`. `_lib.sh` self-locates `MEMORY_DIR` via `BASH_SOURCE` (correct under bash, empty under the Bash-tool's interactive zsh â€” don't sanity-check `. scripts/_lib.sh` from zsh).
- **`archive/` is never read unless the user explicitly asks** â€” and never deleted during a "reorganize memory" pass (audit trail).
- **Frontmatter is the contract** â€” missing/malformed fields break the index; `lint-memory.sh` catches it. `_template/` is excluded from index/lint/regeneration.
- **Working memory is per-project and injected every prompt** â€” keep it short or it bloats every request and busts the cache prefix. Two terminals on the same project both writing `working.md`: last write wins (no merge; acceptable for solo use).
- Slash commands are indexed at session start â€” a new command needs a session restart to autocomplete.
- Oversized hook stdout is routed to a `<persisted-output>` file with a 2KB preview (the hard 10k cap is gone), but a bloated `working.md` still re-ships every prompt.
- **Codex execpolicy** `decision` keyword is `"forbidden"` (not `deny`/`reject`/`block`); invalid values error at `codex exec` time, not parse time. Executor invocation needs `--skip-git-repo-check` when the workdir isn't a git repo.
- **Notion provider:** the API can add/reorder but cannot rename/delete select/status options (rename = add + reorder + leave old vestigial). For `/task`/`/start` to route to Notion, creds + `MEMORY_TASK_PROVIDER=notion` + `NOTION_STATUS_KIND` must be in `~/.zshenv` (NOT `.zshrc`) â€” the Bash tool's non-interactive zsh sources `.zshenv`.
- **Notion refs are ALWAYS the full page UUID, never the 8-char prefix.** A short prefix is *ambiguous* (e.g. `38ef6850` matches both the harness-agnostic task and the delete/trash task) **and** the API rejects it (400 `should be a valid uuid`). `notion.py` guards `get`/`update`/`set_status` (`_require_full_ref`) and fails early with an actionable message; record the full ref (from `taskctl list`/`get`) in `memory.md` and plan `task_ref` frontmatter. The local provider uses slug refs â€” this guard is Notion-only.
- **Plan-mode plan path:** the harness default (`~/.claude/plans/<slug>.md`) conflicts with the canonical path (`projects/<active>/plans/<name>.md`). In plan mode, always write to the memory-system path.
