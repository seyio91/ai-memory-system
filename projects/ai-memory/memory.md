## What It Is
The markdown-only memory system for Claude Code — no vector DB, no daemons, no MCP server, just files + hooks + macOS-`bash`-3.2 shell scripts. This is a **meta-project**: the project being tracked *is* the memory tooling. Single owner (Seyi). Authoritative spec is the repo `README.md` (a lean landing page) plus the deep-reference pages under `docs/` — `docs/harnesses/<name>.md` per harness, mirroring the multi-harness direction.

Three layers (Karpathy LLM-Wiki pattern):
- **Schema** — `identity.md` (hard rules, injected once/session), `~/.claude/CLAUDE.md`. Outranks everything.
- **Wiki** — `domain/*.md` (cross-project) + `projects/*/memory.md` (engagement-specific). Durable.
- **Scratchpad** — `projects/*/working.md`, injected every prompt while non-empty; matures into the wiki via `/promote-memory`.

Moving parts: one `UserPromptSubmit` hook (`~/.claude/hooks/inject_memory.sh`) emits `<memory:*>` blocks; slash commands in `~/.claude/commands/`; scripts in `scripts/` (`_lib.sh`, `regenerate-index.sh`, `lint-memory.sh`, `archive-cleanup.sh`, `new-project.sh`, `memory-pin.sh`, `link-skills.sh`, `executor.sh`, `codex-mem.sh`, the `taskprovider/` package + `taskctl`) with dependency-free bash-3.2 tests under `scripts/tests/`.

## Current State
Functional and in daily use across the real engagements indexed in `index.md`. This checkout is the distributable **source copy**, attached to `origin = seyio91/ai-memory-system` (private). The repo was reconciled onto a canonical GitHub core via the github-core migration (clone-and-swap; live data rsynced in and kept gitignored) — see `archive/plans/github-core-migration.md`. **The engine is now harness-agnostic** (7-phase plan complete 2026-07-06, PRs #16–#22): `install.sh` is a generic manifest-driven installer; `claude`/`codex`/`antigravity` are registered harnesses; the project marker is harness-neutral (`.agents/memory-project`); the executor resolves task/exploration roles from the manifest — see the Architecture Decisions below.

## Architecture Decisions
- **Markdown over a DB** — every editor/diff/grep works; zero infra; git-trackable.
- **Hook-injected, not retrieved** — context arrives in-band. Full payload (identity → project memory → index → working) is injected at SessionStart, on `@memory`, and on the first prompt after compaction; every other prompt gets only the lightweight `<memory:active>` breadcrumb. Domain files are the exception: lazy-read on index trigger-match, never auto-injected.
- **Frontmatter-driven catalog** — `index.md` AUTOGEN block is regenerated from frontmatter (`/reindex`), so it can't lag the files. **Exactly one canonical fence:** `<!-- BEGIN AUTOGEN -->` / `<!-- END AUTOGEN -->` (a `## Projects` roster + a `## Domain` table). The older single combined-table `<!-- AUTOGEN:START -->` / `<!-- AUTOGEN:END -->` block is **retired** — `regenerate-index.sh` never touches it, so any surviving copy silently rots and duplicates the live tables; instances carrying it must delete it. Both scaffold files are excluded from the catalog: the projects loop skips `projects/_template/`, and the **domain loop skips `domain/_template.md`** (its literal `topic: <topic>` frontmatter otherwise leaks a placeholder row) — mirror both guards in any regenerator change, and guard the domain exclusion with a test.
- **Lean index** — `index.md` is a path-less roster; paths are derivable (`projects/<name>/memory.md`, `domain/<topic>.md`), metadata (`tags`/`repo`/`repo_path`, domain `triggers`) stays in the source files. Codex's `AGENTS.md` keeps absolute paths (its shell tool reads by path).
- **Per-session injection markers** (`~/.claude/memory_sessions/<session_id>`, self-expire after 2 days) — once-per-session blocks don't re-inject; concurrent sessions don't collide.
- **Compaction recovery** — `PreCompact` stdout can't inject; the reliable channel is `UserPromptSubmit`. `SessionStart source=compact` writes a per-session sentinel; the next prompt re-injects the full payload once and clears it.
- **Project detection per-prompt** — the hook reads `cwd`, walks up for a harness-neutral `.agents/memory-project` marker (legacy `.claude/memory-project` still read as a fallback at the same level); no marker → no project (memory dormant, no global fallback, so concurrent repos never collide). `.active_project` machinery still exists but is left empty by deliberate decision.
- **Bidirectional repo↔project map** — forward marker `.agents/memory-project` (harness-neutral; `memory-pin.sh` migrates a legacy `.claude` one on pin; `scripts/migrate-marker.sh` bulk-migrates) + reverse `repo`/`repo_path` frontmatter, both written by `/pin` (`memory-pin.sh`). `resolve_repo_path` resolves a checkout per-environment: `repo_path` relative to `projects_root()` (`AI_MEMORY_PROJECTS_ROOT`, default `$HOME/Projects`), git-remote `repo` as portable fallback. `lint-memory.sh` catches reverse-map drift. **`repo_path` is stored relative, never absolute** — the env-var root supplies the prefix so paths resolve identically across host and sandbox. **Self-referential exception:** the `ai-memory` meta-project's checkout *is* the memory tree, so its `repo_path` is the sentinel `$MEMORY_DIR` (and `$MEMORY_DIR/<sub>` is supported) — `resolve_repo_path` and `lint-memory.sh` resolve it against `MEMORY_DIR` rather than `projects_root()`, keeping the committed value machine-independent (each machine sets `MEMORY_DIR` to wherever its tree lives).
- **Distributed cross-project relationships** — a relationship lives in the project where the work *starts* (`## Related Projects`), never in an umbrella. Siblings are **delegated to subagents, never loaded** into the orchestrator thread; plan-set execution pauses at human/CI gates.
- **Orchestrator / Executor / Validator workflow** — Claude main session = orchestrator; an executor runs delegated work; the validator resolves through the **same configured executor** (re-run `executor.sh --which`) as a **separate, fresh invocation** — independence comes from the extra invocation, not a different tool — checking output against the plan's `## Success criteria`. Harness `TaskCreate`/`TaskUpdate` are banned (enforced by `block_task_tools.sh`); `todo.md` is the single source of truth. Archive is never read unless asked. Executors are blocked from any apply/merge/destructive action on running infrastructure.
- **Executor: two roles, manifest-resolved.** `scripts/executor.sh` exposes `--role task` (write-capable) and `--role explore` (read-only), each configured as a `harness[:model]` value (`AI_MEMORY_EXECUTOR_TASK` / `AI_MEMORY_EXECUTOR_EXPLORE`, falling back to the legacy single `AI_MEMORY_EXECUTOR`, then `claude-subagent`). A harness resolves **through its `harnesses/<name>/manifest` `exec_*` block** — no per-harness code (the old codex special-case is gone): `exec=subagent` → subagent plane; else `exec_cmd` (task) / `exec_readonly` (explore), gated on `exec_probe`. A harness with no read-only mode is **skipped for `explore`** (degrades to the Claude Explore agent), never run write-capable. Model threads via `exec_model_flag`; the subagent plane surfaces `subagent:<model>`. `--which` prints `subagent[:model]` | `cli:<name>`; `--run` execs or bounces to the subagent (exit 3). **Which executor a given instance uses is config-driven, never recorded here** (set the vars in gitignored `config.local.sh`; the orchestrator discovers via `executor.sh --role <r> --which`). The codex deny-rules file is optional codex-only hardening; the no-merge deny-list is restated in every delegation prompt regardless of executor.
- **Task-provider layer + `/task`/`/start`** — pluggable capture/track/execute subsystem; stdlib-only Python `scripts/taskprovider/` (generic registry: env value `MEMORY_TASK_PROVIDER` *is* the provider module → no factory edit to add a backend), `providers/local.py` (flat `tasks/<slug>.md`) + `providers/notion.py` (urllib-only). Backend is a projection + intake queue, push-dominant, never bidirectional. `scripts/taskctl` bash wrapper; `/task` (add|list|done|archive|show), `/start <ref>` (runs the brainstorm gate → scaffolds a plan into the task's own project).
- **Brainstorming skill (gated Tier-3 design pass)** — runs the collaborative design pass only for Tier-3 feature tasks with open design questions; silent on Tier 1/2 and settled Tier-3. Terminal handoff is `/new-plan`, folding the approved design into the plan's Goal/Success criteria/Design/Risks. Orchestrator-only, no separate committed spec.
- **Task Contract — success criteria** — plan-tier tasks carry explicit checkable success criteria (the Validator's contract). Policy in `identity.md` → `### Task Contract`; enforcement is template-only (`/new-plan` scaffold). A criteria-less plan is a process failure (draft them first), not a license to invent a bar.
- **Multi-git-provider support** — projects span GitHub, Bitbucket Cloud, Azure DevOps. Provider is inferred from each project's `repo` remote host (no `git_provider` field). PR/repo tooling routes by provider: `gh` / `bkt` skill / `az repos`. The no-merge guardrail extends to all three (Azure merge = `az repos pr update --status completed`).
- **Skills owned by memory, symlinked into harnesses** — authored skills live in `skills/<name>/` and remote skills materialize in `.skill-cache/<name>/`; each runtime's skills dir holds per-skill symlinks back into those stores. Sync via `scripts/link-skills.sh`.
- **Skill subsystem — validation + on-request self-rating; no write boundary.** Skills carry no write-boundary declaration: `metadata.tier` and the detective git-diff boundary check (the two in-session hooks + `skill-boundary-check.sh`) were removed — the mechanism was detective-only, overlapped execpolicy and the Validator, and didn't fit remote skills (external repos can't carry the field). The floors that remain: **execpolicy** is the destructive-class floor, and the **Validator** catches a skill that mutated something it shouldn't (against the plan's success criteria). `validate-skills.sh` statically checks the schema (name/description, size, placeholders); `new-skill.sh` / `install-skill.sh` author / intake to it. First-party workflow skills carry an **on-request** self-rating block (skill friction, a separate axis from the Validator) — loop membership is **marker-derived**, not a static list; this stays intact. Authoring rule: every script action keeps a hand-editable equivalent (Two-Path principle). Historical design record in `archive/plans/skill-subsystem.md` (boundary apparatus removed by `archive/plans/remove-metadata-tier.md`).
- **Skill source model** — skills now have one meaningful axis: **source**. Authored skills live in `skills/`, which is per-instance and gitignored except `.gitkeep`; the old generic/local authored split is retired, so there is no tracked in-tree authored skill store. Remote skills are declared in the per-instance root `skills.toml`, seeded from the tracked `skills.toml.example` catalog on install, fetched into the gitignored `.skill-cache/`, and pinned by a lockfile: *referenced, not forked*. Sharing a skill means publishing it to a remote repo and adding it to the catalog/manifest. Stateful skills persist local data in a gitignored `.skill-data/<name>/` root (`AI_MEMORY_SKILL_DATA`), decoupled from the ephemeral skill dir so remote-referenced skills don't lose state on re-resolve. Enumeration is centralized in `_lib.sh:list_skill_dirs` over the two roots (`skills/` + `.skill-cache/`; override via `AI_MEMORY_SKILL_ROOTS`). `install-skill --remote --save` writes the manifest, `resolve-skills.sh` (+ `sync-system.sh --update`) materialize it, `list-skills` shows provenance. Full reference in `docs/harnesses/claude.md` → Skill Source.
- **Knowledge lifecycle** — per-project `working.md` → `domain/<topic>.md` or project `memory.md` Decisions Log (via `/promote-memory`) → packaged skill (manual graduation when a domain file grows past ~50 lines or holds a coherent reusable pattern).
- **Derived state snapshot (`/state`)** — `scripts/regenerate-state.sh` projects a single on-demand "In Flight" table across all projects (`project | last touched | current goal | open todos`), every column derived (last-touched = newest file mtime — *not* `git log`, since most project trees are gitignored; current goal from `## Current Goal`; open-todo count from `todo.md`). A projection like `index.md` (can't drift), but **on-demand only — never auto-injected**: it surfaces *that* sibling work exists without loading sibling `memory.md` (delegate-don't-load). Output `state.md` is gitignored. Exposed via `/state`.
- **Project categories + activity report** — projects carry an optional `category:` (client/group) in `memory.md` frontmatter. **Per-instance personal data:** the engine supports the field, but real project `memory.md` is gitignored so values never enter git history (the tracked `ai-memory` meta-project stays category-less; `_template` ships a commented placeholder). Set via `/pin --category`, `/new-project`, or hand-edit. Powers category-grouped `/state` (uncategorized last) and **`/activity`** (`regenerate-activity.sh`): the plans *created* in a window (default 30d), grouped by category, scanning live **and** archived plans — the billable/reviewable unit is a **plan** (`created:`), so the report is decoupled from the task backend. Output `activity.md` gitignored, on-demand. One flat category per project.
- **Wikis document existing system components, not plan references** — `wikis/` is for durable docs of what the system *is*; comparison/triage/adoption-backlog docs are plan inputs and get archived to `archive/wikis/` once the plan using them ships (the POS triage docs were archived when state-snapshot landed).
- **This meta-project is tracked in-repo** — unlike every other project (external repo + client data → gitignored), `projects/ai-memory` documents *this* repo and carries no secrets, so its durable artifacts are version-controlled via a `.gitignore` carve-out: **tracked** = `memory.md`, `plans/`, `todo.md`, **and `archive/`** (past work stays visible in git — the memory system simply never *loads* `archive/`, a runtime convention, not a tracking one); **ignored** = only `working.md` (per-machine live scratch). Repo is private, so internal repo/org names are acceptable to track. Consequence: routine `memory.md`/`todo.md` edits and plan-archival become git changes — **commit these directly to `main` (no PR)**; reserve a branch + PR for new features / substantive code changes.
- **Per-environment config loader; instance config never in tracked memory** — `_lib.sh` and `taskctl` source a gitignored `config.local.sh` next to the tree; ship `config.local.sh.example`. Because the same core/memory is shared across multiple instances, **machine/instance-specific choices live in `config.local.sh`, not in this tracked `memory.md`** — projects root (`AI_MEMORY_PROJECTS_ROOT`), executor (`AI_MEMORY_EXECUTOR`), task provider (`MEMORY_TASK_PROVIDER`), etc. Scripts read them by sourcing `_lib.sh`; the orchestrator reads runtime-derived ones via their tool (e.g. `executor.sh --which`). The committed default root is `$HOME/Projects`.
- **Harness-agnostic, manifest-driven engine** — no harness is privileged. Each lives under `harnesses/<name>/` and is registered by a declarative `manifest` (data, never sourced) with two capability faces: **deliver** (install surfaces — `archetype` `hook`|`file`, `format` `xml`|`md`, `context_target`/`skills_dir`/`commands`…) and **execute** (`exec_*` — the executor contract). `install.sh` is the generic, agent-runnable engine (`--harness <name>` | auto-detect | `--list`): resolve → validate manifest → archetype driver (`scripts/drivers/{hook,file}.sh`) → deliver commands/skills/agents → optional per-harness override. Content selection lives once in `scripts/content-core.sh` (+ `formatters/{xml,md}.sh`); the file-archetype md context builds once via `scripts/build-context-md.sh` (Codex's `codex-mem.sh` calls it). **Registered today:** `claude` (hook/xml), `codex` (file/md), `antigravity` (hook/xml — live injection + enforced executor; see the Antigravity decision below). Adding a harness is a manifest, not engine code. Full design + phase record in `archive/plans/make-memory-engine-harness-agnostic.md`.
- **Codex adapter** — `codex-mem.sh` (a `file`-archetype launch wrapper) regenerates `~/.codex/AGENTS.md` on every launch via the shared `build-context-md.sh` (no hooks in Codex; only the injection mechanism differs). `--executor` = `exec --sandbox workspace-write` network-on; `--executor-bare` strips the memory merge (~13k tokens) for read-only review subagents. Mid-session memory edits don't appear until relaunch. Codex is now a fully manifest-registered harness (install target + executor), not a bespoke bolt-on.
- **Antigravity harness — hook archetype, live injection + enforced executor** — Antigravity (`agy`) is a genuine `hook` archetype, not a materialized file: memory injects **live per model call** via a `PreInvocation` hook in `~/.gemini/config/hooks.json` (built from the shared `content-core.sh` + `xml.sh` — the same `<memory:*>` payload as Claude; `invocationNum==0` → full, else breadcrumb). Its hook payload carries **no workspace handle**, so `agy.sh` resolves the active project at launch and exports it (`AI_MEMORY_PROJECT`/`MEMORY_DIR`) for the hook to read; the static workflow-rules base is a hand-owned `~/.gemini/config/AGENTS.md` (agy has no `AGENTS.local.md`). It is also the **2nd enforced executor after codex**: a `PreToolUse` guard (`AI_MEMORY_ROLE`-gated, so interactive `agy` stays unguarded) enforces the shared `scripts/deny-list.txt` for both roles and a read-only **allowlist** for `explore` — giving Antigravity a real `exec_readonly` instead of degrading to Claude Explore. The deny-list is now a **shared shipped artifact** (seeding a future manifest `guard` capability). It also ships a **memory-aware statusline** (`statusline.sh`, registered via `settings.json → statusLine` by an idempotent install merge that preserves existing keys) surfacing the active project + folder + branch/model/context/agent-state; emoji glyphs by default (matching Claude — `USE_NERD_FONTS=true` opts into Nerd Font). Full design + phase record in `archive/plans/antigravity-hook-archetype.md` and `archive/plans/antigravity-statusline.md`.
- **Non-goals:** no `PreToolUse` deny enforcing `todo.md` (documented convention only). ~~No bootstrap script.~~ **Reversed (Phase 5, harness-agnostic work):** `install.sh` IS the agent-runnable, manifest-driven bootstrap (`--harness <name>` | auto-detect | `--list`) — an agent inside any harness runs it to wire that harness up; README demoted to backup.

## Known Constraints / Gotchas
- **This repo IS under git, but personal content is `.gitignore`d.** Authoritative list is `.gitignore` — verify there. Tracked (distributable source): `scripts/`, `harnesses/`, `agents/`, `install.sh`, `README.md`, `identity.md`, `config.local.sh.example`, `skills.toml.example`, `projects/_template`, `domain/_template.md`, and `skills/.gitkeep`. Ignored (personal/runtime): `.claude/`, `.sessions/`, `/index.md`, `/domain/*` (except `_template.md`), `/projects/*` (except `_template` **and `ai-memory`**), `/tasks/`, `/archive/`, `/config.local.sh`, `/skills/*` (authored per-instance skills; `.gitkeep` tracked), `/.skill-cache/` (materialized remote skills), `/docs/superpowers/`. Live per-engagement memory never gets committed **except this `ai-memory` meta-project**. Never commit secrets.
- **Scripts target macOS `bash` 3.2** — no `mapfile`, no associative arrays. Resolve the tree via `MEMORY_DIR`. `_lib.sh` self-locates `MEMORY_DIR` via `BASH_SOURCE` (correct under bash, empty under the Bash-tool's interactive zsh — don't sanity-check `. scripts/_lib.sh` from zsh).
- **`archive/` is never read unless the user explicitly asks** — and never deleted during a "reorganize memory" pass (audit trail).
- **Frontmatter is the contract** — missing/malformed fields break the index; `lint-memory.sh` catches it. `_template/` is excluded from index/lint/regeneration.
- **Working memory is per-project and injected every prompt** — keep it short or it bloats every request and busts the cache prefix. Two terminals on the same project both writing `working.md`: last write wins (no merge; acceptable for solo use).
- Slash commands are indexed at session start — a new command needs a session restart to autocomplete.
- Oversized hook stdout is routed to a `<persisted-output>` file with a 2KB preview (the hard 10k cap is gone), but a bloated `working.md` still re-ships every prompt.
- **Codex execpolicy** `decision` keyword is `"forbidden"` (not `deny`/`reject`/`block`); invalid values error at `codex exec` time, not parse time. Executor invocation needs `--skip-git-repo-check` when the workdir isn't a git repo.
- **Notion provider:** the API can add/reorder but cannot rename/delete select/status options (rename = add + reorder + leave old vestigial). For `/task`/`/start` to route to Notion, creds + `MEMORY_TASK_PROVIDER=notion` + `NOTION_STATUS_KIND` must be in `~/.zshenv` (NOT `.zshrc`) — the Bash tool's non-interactive zsh sources `.zshenv`.
- **Notion refs are ALWAYS the full page UUID, never the 8-char prefix.** A short prefix is *ambiguous* (e.g. `38ef6850` matches both the harness-agnostic task and the delete/trash task) **and** the API rejects it (400 `should be a valid uuid`). `notion.py` guards `get`/`update`/`set_status` (`_require_full_ref`) and fails early with an actionable message; record the full ref (from `taskctl list`/`get`) in `memory.md` and plan `task_ref` frontmatter. The local provider uses slug refs — this guard is Notion-only.
- **Plan-mode plan path:** the harness default (`~/.claude/plans/<slug>.md`) conflicts with the canonical path (`projects/<active>/plans/<name>.md`). In plan mode, always write to the memory-system path.
