## What It Is
The markdown-only memory system for Claude Code ‚Äî no vector DB, no daemons, no MCP server, just files + hooks + macOS-`bash`-3.2 shell scripts. This is a **meta-project**: the project being tracked *is* the memory tooling. Single owner (Seyi). Authoritative spec is the repo `README.md` (a lean landing page) plus the deep-reference pages under `docs/` ‚Äî `docs/harnesses/<name>.md` per harness, mirroring the multi-harness direction.

Three layers (Karpathy LLM-Wiki pattern):
- **Schema** ‚Äî `identity.md` (hard rules, injected once/session), `~/.claude/CLAUDE.md`. Outranks everything.
- **Wiki** ‚Äî `domain/*.md` (cross-project) + `projects/*/memory.md` (engagement-specific). Durable.
- **Scratchpad** ‚Äî `projects/*/working.md`, injected with the full payload at session start (re-injected on `@memory`/after compaction) while non-empty ‚Äî **not** every prompt; matures into the wiki via `/promote-memory`.

Moving parts: one `UserPromptSubmit` hook (the shared `scripts/hooks/inject.sh`, format-parameterized) emits `<memory:*>` blocks; slash commands in `~/.claude/commands/`; scripts in `scripts/` (`_lib.sh`, `regenerate-index.sh`, `lint-memory.sh`, `archive-cleanup.sh`, `new-project.sh`, `memory-pin.sh`, `link-skills.sh`, `executor.sh`, `codex-mem.sh`, the `taskprovider/` package + `taskctl`) with dependency-free bash-3.2 tests under `scripts/tests/`.

## Current State
Functional and in daily use across the real engagements indexed in `index.md`. This checkout is the distributable **source copy** and the dev instance: `origin = seyio91/ai-memory-system` (private, headed public), `AI_MEMORY_CHANNEL=dev` so a sync never flips it onto a tag.

The engine is **harness-agnostic** (`install.sh` is a generic manifest-driven installer; `claude`/`codex`/`antigravity` are registered harnesses; the marker is `.agents/memory-project`; the executor resolves roles from the manifest) and **versioned**: consumer instances sync to stable `v*` tags, never to a moving `main`. Latest tag `v1.1.0`; `v1.0.0` is a trap tag (see Gotchas). Design records live in Architecture Decisions below.

## Architecture Decisions
- **Markdown over a DB** ‚Äî markdown files are the durable store; normal editor, diff, grep, and git workflows are the database interface.
- **Hook-injected, not retrieved** ‚Äî context is delivered in-band: full payload on session start, `@memory`, and first post-compaction prompt; breadcrumbs otherwise; domain files stay lazy-loaded.
- **Frontmatter-driven catalog** ‚Äî `index.md` is regenerated from frontmatter inside the `<!-- BEGIN AUTOGEN -->` / `<!-- END AUTOGEN -->` fence; retired fences are ignored, and scaffold files stay out of the catalog.
- **Lean index** ‚Äî `index.md` lists names and summaries only; paths are derivable, while repo metadata and domain triggers remain in the source files.
- **No per-session marker; `SessionStart` is the once-per-session guarantee** ‚Äî initial full injection relies on the hook event itself; only compaction recovery uses a per-session sentinel.
- **Compaction recovery** ‚Äî compaction rehydration is a sentinel handoff from `SessionStart source=compact` to the next `UserPromptSubmit`, which emits the full payload once.
- **Project detection per-prompt** ‚Äî active project resolution walks up from `cwd` to `.agents/memory-project`; no marker means memory is dormant, with legacy `.claude/memory-project` read only as fallback.
- **Bidirectional repo‚Üîproject map** ‚Äî `.agents/memory-project` is the forward map and `repo`/`repo_path` frontmatter is the reverse map; `repo_path` is relative except the `$MEMORY_DIR` self-reference for `ai-memory`.
- **Distributed cross-project relationships** ‚Äî related-project knowledge lives where the work starts; sibling work is delegated rather than loaded, and plan-set execution pauses at human or CI gates.
- **Orchestrator / Executor / Validator workflow** ‚Äî the main session orchestrates, executors implement, and the read-only validator role checks plan success criteria; `todo.md` remains the plan tracker and destructive executor actions stay forbidden.
- **Executor: two roles, manifest-resolved.** `executor.sh` resolves `task` and `explore` through harness manifests and config-local executor settings; read-only roles never fall back to write-capable execution.
- **Task-provider layer + `/task`/`/start`** ‚Äî tasks are a pluggable, push-dominant projection over local or Notion backends; `/task` captures intent and `/start` turns it into a plan.
- **Task `summary` is a capped thin record; long-form lives in the tree by pointer** ‚Äî task summaries are capped at 500 characters, while detailed pre-plan material lives as named investigations in the project tree; Notion page bodies are outside the provider contract.
- **`add_progress` stays an unwired no-op ‚Äî settled WON'T BUILD (2026-07-20).** `contract.py:117` is the only non-abstract method on the provider ABC, and that asymmetry is deliberate: the concrete `return None` is what lets a new backend opt *in* rather than be forced to stub it. **Do not "clean up" the unused method** ‚Äî it is load-bearing. Not wired to `/checkpoint` because the two don't line up: `add_progress(ref, note)` needs a task ref, while `/checkpoint` is scoped to a project and a session and may cover zero, one, or several tasks. Compounding it, a progress note is neither a capped `summary` nor a page body (which is outside the contract), so Notion has no defined home for one, and checkpoints are high-volume enough to have needed `/checkpoint-archive` ‚Äî pushing each one outward would be one-directional noise with no reconciliation path. If ever revisited, the live plan's `task_ref` frontmatter is the strongest ref source, since `/plan-archive` already relies on that seam.
- **Brainstorming skill (gated Tier-3 design pass)** ‚Äî only Tier-3 feature work with open design questions runs the brainstorming pass; its output is folded into the plan, not a separate spec.
- **An investigation is an artifact; a brainstorm is an activity** ‚Äî investigations are optional long-form findings artifacts consumed by tasks and `/start`; brainstorms are the dialogue that produces the plan, and investigations archive with their task.
- **Task Contract ‚Äî success criteria** ‚Äî plan-tier tasks must carry explicit, checkable success criteria before execution; the validator checks exactly that contract.
- **Multi-git-provider support** ‚Äî repository provider is inferred from the remote host, and PR/repo tooling routes through GitHub, Bitbucket, or Azure DevOps while preserving the no-merge guardrail.
- **Skills owned by memory, symlinked into harnesses** ‚Äî skills live in memory-owned stores and runtime skill directories receive symlinks from `scripts/link-skills.sh`.
- **Skill subsystem ‚Äî validation + on-request self-rating; no write boundary.** Skills have schema validation and optional self-rating, but no declared write-boundary mechanism; execution policy and the validator are the enforcement layers.
- **Skill source model** ‚Äî authored skills are local, remote skills are manifest-declared and lock-pinned, skill state lives under a separate data root, and enumeration is centralized across source roots.
- **Doc-vs-code gate ‚Äî the `docs/scripts.md` env-var table is machine-checked** ‚Äî documented script environment variables are checked against code and transitive sourcing paths; the gate is scoped to symbol drift, not semantic prose.
- **A control is not trusted until it has been watched to fail.** Controls must be mutation-tested or otherwise observed failing on known defects before their green result is trusted.
- **Executable tests cannot gate a command whose logic is prose ‚Äî only a live exercise can.** Slash commands are natural-language instructions an agent follows, so no test in `scripts/tests/` executes them. `/plan-archive` shipped with Step 5 saying "skip to Step 8" twice, which skips Step 7 ‚Äî the plan move itself ‚Äî breaking the *default* path (no linked investigation) while the full suite stayed green and its rule-10 lint check passed mutation testing in both directions. Live-exercise every prose command on its **most common** path, not just its interesting one.
- **Knowledge lifecycle** ‚Äî working memory matures into domain/project memory and, when reusable enough, into packaged skills.
- **Derived state snapshot (`/state`)** ‚Äî `/state` is an on-demand derived projection of active work across projects and is never auto-injected.
- **Project categories + activity report** ‚Äî projects may carry one flat category for grouped `/state` and `/activity` reporting; the report groups plans by creation window.
- **Durable memory names artifacts; it does not store their paths** ‚Äî durable memory references artifacts by name, not path; only live machine-consumed fields may hold paths.
- **Wikis document existing system components, not plan references** ‚Äî `wikis/` records durable system components; comparison, triage, and backlog material belongs to plan inputs and archives after use.
- **This meta-project is tracked in-repo** ‚Äî `projects/ai-memory` is the tracked dogfood project for this repo, with durable memory artifacts versioned and `working.md` left local-only.
- **Commit route is decided by *what* changed, not how big the change is: memory tree vs shipped system.** **Housekeeping** ‚Äî anything under `projects/` (todos, plans, investigations, `memory.md`, archive moves) ‚Äî is bookkeeping *about* work, changes no behavior, and goes **directly to `main`, no PR, no CI**. **System changes** ‚Äî `scripts/`, `harnesses/`, hooks, `install.sh`, `docs/`, `.github/`, skills ‚Äî ship to consumers and require a **branch + PR**, regardless of how small the diff looks. A one-line hook edit is a system change; archiving five plans is not. Size is not the axis, and "it's only a doc" is not an exemption: `docs/` is shipped. **The tracked `domain/` files are system changes and need a branch + PR** (decided 2026-07-18, after routing one to `main` and reverting it). `/domain/*` is gitignored *except* `_template.md`, `agent-tooling.md`, `codex.md` (`.gitignore:29-32`) ‚Äî those three ship in every release tag, so **users read them**, and content a user consumes is reviewable regardless of whether anything executes. The axis is **does it reach users**, not "does it run": a wrong domain entry teaches a wrong model to every consumer, which is exactly the harm review exists to catch. Untracked `domain/` files are personal and need no route at all.
- **Open-sourcing: remove content going forward, never scrub** ‚Äî the public transition removes live personal/client content from current tracked files but does not rewrite published history or archive audit trail.
- **Per-environment config loader; instance config never in tracked memory** ‚Äî instance choices live in gitignored `config.local.sh`, with tracked examples only; scripts and orchestration read runtime config from there.
- **Harness-agnostic, manifest-driven engine** ‚Äî harnesses are registered by manifests with delivery and execution faces; `install.sh` drives shared installation through manifest data and common content formatting.
- **Codex adapter ‚Äî hook-base, chunked (the Antigravity model)** ‚Äî Codex uses a hand-owned static base plus live chunked hook injection for dynamic memory; `codex-mem.sh` remains only as the executor wrapper.
- **Antigravity harness ‚Äî hook archetype, live injection + enforced executor** ‚Äî Antigravity uses hook-time XML memory injection, an enforced guarded executor mode, and its own memory-aware statusline.
- **Hook layer ‚Äî declarative, manifest-driven, shared where isomorphic** ‚Äî hook roles are declared in manifests; Claude and Codex share behavioral hook scripts where their contracts match, while harness-specific registration remains allowed.
- **Hook layer ‚Äî share the isomorphic, special-case the divergent; don't abstract over a bounded harness set.** Shared hook abstractions stop at genuinely common contracts; divergent harnesses keep thin local paths rather than forcing a universal adapter.
- **Versioned release channel ‚Äî a git tag IS the release; no zip, no separate engine root** ‚Äî consumers sync to stable tags or dev refs; migrations are forward-only/idempotent and releases are guarded, resumable tag cuts.
- **Release automation ‚Äî SHIPPED; changelog is written per-PR, never drafted at release time.** Each PR drops a `changelog.d/<id>.<kind>.md` fragment (`breaking`/`feature`/`fix`/`upgrade`); `assemble-changelog.sh` turns them into a CHANGELOG section and derives the version from the kinds. `release-pr.yml` opens/refreshes the Release PR, merging it is the human gate, and `release-publish.yml` then runs `release.sh --publish` to tag and publish. **The rejected alternative matters:** having a model write the changelog *in CI* was considered and refused ‚Äî by release time the *why* is gone, so it reconstructs intent from commit subjects, which is exactly where hallucinated release notes come from. Model drafts the fragment while the reasoning is live; the suite enforces the invariant (`test_upgrading_doc.sh`: every `migrations/<v>-*.sh` has a matching `UPGRADING.md` section). A model is deliberately kept **off** the CI critical path ‚Äî no API key in CI, no nondeterminism in a publish step. Also decided: **build no changelog/upgrade-note skills** ‚Äî a fragment is a by-product of the PR step, not a capability. Full reasoning in the archived `release-automation` plan + wiki.
- **CI skips the suite for the four root community files, via a gate job ‚Äî never `paths-ignore` (decided 2026-07-20, PR #91).** Exempt: `README.md`, `CONTRIBUTING.md`, `SECURITY.md`, `CODE_OF_CONDUCT.md`. Verified to have zero test surface ‚Äî `lint-memory.sh` globs only `domain/*.md` and `projects/*/‚Ä¶`, `check-docs` parses only `docs/scripts.md`, and the tests matching `README` use it as a sandbox fixture filename. **`docs/**` is deliberately NOT exempt**: `docs/scripts.md` is machine-checked and `docs/` ships in every release tag. Three constraints hold this shape and must survive any future edit: (1) **gate job, not `paths-ignore`** ‚Äî the latter skips the workflow entirely, reports *no status*, and deadlocks a required check under branch protection (a stated goal); (2) **fail closed** ‚Äî empty diff or failed fetch runs the suite; (3) **anchored match** (`grep -vxE`) ‚Äî unanchored, `docs/README.md` becomes exempt and the suite is skipped on a real docs change; this was mutation-tested and is the failure mode to check first. Growing the exemption list requires re-running the verification, because a rotting list fails *open*. **My first recorded version of this decision (same day) said path filters were a deliberate non-goal ‚Äî that was superseded within the hour after the user pushed back twice; the "markdown is not inert" argument is about `docs/`, and does not defend running a suite that provably reads nothing.**
- **The tests badge must carry `?event=pull_request` (fixed 2026-07-20, PR #89).** A workflow badge with no query params resolves to the latest run on the **default branch**, but `tests.yml` fires on `pull_request` only (#78 removed the push trigger), and a PR run's `head_branch` is never `main`. The badge therefore froze on run `29652600768` ‚Äî a billing refusal from 2026-07-18 ‚Äî and no future run could supersede it. **Removing a push trigger silently freezes that workflow's badge forever;** re-check the badge whenever a trigger changes.
- **Non-goals:** no runtime deny hook enforces `todo.md`; `install.sh` is the supported agent-runnable bootstrap.

## Known Constraints / Gotchas
- **This repo IS under git, but personal content is `.gitignore`d.** The tracked source tree is distributable, personal/runtime memory is ignored, and only the `ai-memory` meta-project is intentionally versioned.
- **Scripts target macOS `bash` 3.2** ‚Äî scripts must remain compatible with Bash 3.2 and resolve the memory tree through `MEMORY_DIR`.
- **`archive/` is never read unless the user explicitly asks** ‚Äî archive content is audit trail, not working context, and is not deleted during reorganization.
- **Frontmatter is the contract** ‚Äî catalog, lint, and regeneration depend on valid frontmatter; templates are excluded.
- **Working memory is per-project; its content rides the full payload injected once per session (and on `@memory`/post-compaction), NOT every prompt.** Keep `working.md` short; concurrent sessions on the same checkout are last-write-wins.
- Slash commands are indexed at session start ‚Äî a new command needs a session restart to autocomplete.
- Oversized hook stdout is routed to a `<persisted-output>` file with a 2KB preview, but bloated `working.md` still expands full-payload injection.
- **For a security control, a green test suite is where validation STARTS, not ends.** Security controls need adversarial validation beyond the normal suite; false-positive fixes must be checked for false negatives.
- **A test file the runner's glob doesn't reach is silently ungated ‚Äî the suite reports green and proves nothing.** New test locations must be wired into the runner or they are not part of the gate.
- **A `set -e` test that dies mid-run reports a bare rc=1 with no reason, no failing assertion, and no summary ‚Äî every assertion after the death point is ungated while the file still *looks* like one failing test.** Wrap fallible checks in `set +e` / `rc=$?; set -e` so failures reach the reporter. Corollary: an assertion that hardcodes a value the code derives from config rots the moment config changes ‚Äî derive the expectation from the same source (`test_install_harness.sh` matched 0 entries and hid 106 of 143 assertions from `742f083` until 2026-07-18). "Fails at HEAD too" establishes *not mine*, never *not related* ‚Äî always trace the cause.
- **A recorded measurement is where planning STARTS, not ends ‚Äî re-measure before you act on it, especially when it is the fact that justifies the work.** Planning baselines must be re-derived before execution, and each proposed control must be matched empirically to the bug class it claims to catch.
- **Codex execpolicy** `decision` must be `"forbidden"`; invalid values fail at `codex exec` time, and executor runs outside git repos need `--skip-git-repo-check`.
- **ROOT-CAUSED (2026-07-18): the phantom-empty `ls` was never TCC ‚Äî it is `rtk ls` returning nothing on a non-`en_US` locale.** `rtk 0.38.0`'s `ls` exits **0 with empty stdout and empty stderr** whenever `LANG`/`LC_ALL` is anything other than `en_US.*` or `C` ‚Äî verified on freshly-created dirs in both orders (`nl_NL`, `en_GB`, `de_DE`, `fr_FR` ‚Üí empty; `en_US`, `C` ‚Üí correct), so it is locale, not caching or ordering. This box runs `LANG=nl_NL.UTF-8` (hence the Dutch `mrt`/`jun` in `ls -l`), and the Claude hook rewrites bare `ls` ‚Üí `rtk ls`, so **every bare `ls` silently returned empty**. Note `en_GB` also fails: the split is `en_US`/`C` vs everything else, not English vs non-English. Fixed here by `export LC_ALL=en_US.UTF-8` in `~/.zshenv` ‚Äî **not** `LC_ALL=C` (byte-mode would mangle the non-ASCII this tree is full of), and **not** `config.local.sh`, which no shell profile ever sources. Escape hatch if it recurs: `rtk proxy <cmd>` bypasses the filter. Blamed on TCC for weeks and "fixed" by relocating the tree, which of course changed nothing ‚Äî **a failure that is silent AND exits 0 will be misattributed to whatever looks suspicious nearby; bisect the environment instead of trusting the plausible story.**
- **`ls`/glob output is not evidence for a count or an absence claim ‚Äî corroborate with `find` or `git ls-files`.** Independent of the rtk bug above, this is the durable habit: the failure reads as a real result, so it fabricates confident wrong conclusions in both directions ‚Äî absent-when-present *and* present-when-absent.
- **A truncating pager on test output hides scope, and the per-file listing is not the coverage signal ‚Äî the summary counter is.** `run-tests.sh` emits >60 lines, so a habitual `| tail -60` silently drops the alphabetically-earliest files (16 of 48 on 2026-07-18) and the surviving listing looks complete. Reconcile the printed file count against the `tests: N passed` total before calling a run full; the banner absence (`SELECTED RUN`, `NOT A FULL RUN`, `UNMAPPED`) is necessary but not sufficient.
- **`link-skills.sh` only ever visits skills that still exist, so a link whose source vanished is never revisited** ‚Äî a rename or a tree move strands it indefinitely (`dashboarding`, `tempo` survived both, dangling from 30 jun to 18 jul 2026). The prune pass therefore matches on **shape** (dangling + target directly under `skills/` or `.skill-cache/` + basename equals link name) rather than against the *current* store roots, because a moved tree leaves links pointing at a root that is no longer configured.
- **A git worktree on this self-referential meta-project forks the memory tree, but `MEMORY_DIR` stays pinned to the main checkout.** In `ai-memory`, plan/todo/memory bookkeeping stays canonical in main; worktrees are for engine code only unless isolated with their own `MEMORY_DIR`.
- **The `1/1` no-chunking default is fine on `copilot` and `antigravity`; only Claude needed chunking (validated live 2026-07-18).** Neither manifest declares `session_chunks`/`inject_chunks`, so both deliver the whole base as one message ‚Äî copilot 52,780 chars (its `md` path renders an extra `domain` section) and antigravity 46,284 (`xml`) ‚Äî and both were confirmed to load after a `/clear`. The missing keys are therefore *not* a latent copy of the Claude bug: Claude's ~10,000-char `additionalContext` cap is the outlier, not the norm. Caveat: point-in-time at ~50KB, and nothing emits a marker if either later crosses a cap as memory grows ‚Äî that failure would be silent, the same shape as Claude's. Verify the payload's **tail** (`working` renders last in both formats), never just that memory "appeared": Claude's bug still showed an `identity.md` preview, so a presence check passes while four of five sections are missing.
- **Registration order is not delivery order ‚Äî Claude runs same-event hook entries concurrently and concatenates by completion.** Codex delivers in registration order; Claude does not (observed 2026-07-18: entries registered 1..12 arrived 2,3,4,1,5). Any payload fanned across N hook entries must be self-ordering ‚Äî slices carry a `<memory:chunk index of>` envelope ‚Äî because line-boundary slicing means an out-of-order chunk bisects a content block. A per-harness behaviour verified on one harness must not be assumed for another.
- **Codex `exec` (headless) fires `SessionStart(source=startup)` AND injects its `additionalContext` into model context ‚Äî verified by live probe (2026-07-16, codex 0.144.4; floor 0.135.0).** Startup injection is proven for headless Codex; compaction survival is still unproven and must not be assumed.
- **Notion provider:** Notion can add/reorder select/status options but cannot rename/delete them; non-interactive routing requires credentials and provider env in `.zshenv`.
- **Notion refs are ALWAYS the full page UUID, never the 8-char prefix.** Short refs are ambiguous and API-invalid; local-provider slug refs do not change this Notion rule.
- **Plan-mode plan path:** plan-mode artifacts belong under `projects/<active>/plans/<name>.md`, not the harness default plan directory.
- **`config.local.sh` beats the environment.** Values exported from gitignored `config.local.sh` override process env, so instance channel/provider/executor changes must be made there.
- **`git tag -a -m` strips every line starting with `#`** ‚Äî annotated release tags must preserve markdown messages with verbatim cleanup/file input. `git tag -l -n` reformats the message and hides the loss; only `git cat-file tag` shows the raw object.
- **`--to <branch>` checks out that ref as-is and does not fast-forward it.** Use an up-to-date remote ref when dogfooding or syncing to a branch.
- **`v1.0.0` is a trap tag ‚Äî never point an instance at it.** It predates `identity.md` untracking and can clobber personalized identity data; `v1.1.0` is the first safe release tag.
- **Docs rot within hours when phases ship serially.** Runtime changes require rereading command docs as well as reference docs.
- **A tracked `identity.md` bricks the release channel.** User-editable files must not be tracked; untracking a previously tracked file can delete local data during checkout transitions.
- **A git tag ships a whole tree, and there is no build step to filter it.** Under tag-based distribution, tracking a file publishes it in every release; `.gitattributes export-ignore` is no defence ‚Äî `git archive` honours it, `git clone`/`git checkout <tag>` do not.
- **A GitHub force-push does not remove anything.** GitHub-controlled PR refs can preserve scrubbed history; deleting/recreating the remote is the only complete removal path.
- **`domain/<topic>-cache/` subdirectories are invisible to both the catalog and git.** `regenerate-index` globs top-level `domain/*.md` only (`-maxdepth 1`), and `/domain/*` is gitignored ‚Äî so a path-addressed cache like `domain/terraform-module-cache/` (one file per module, keyed on short name) is local, untracked, and looked up by direct path. Never reindex after writing one; it will not appear.
