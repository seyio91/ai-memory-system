**[2026-06-29]** Codex execpolicy (`~/.codex/rules/default.rules`, the executor guardrail) is **deny-override, not longest-match** — when several `prefix_rule`s match a command, `forbidden` always wins regardless of specificity. Verified with `codex execpolicy check --rules <file> <cmd…>` (non-mutating; use it to test rules): a specific `gh pr merge→forbidden` beats a broad `gh→allow`, AND a broad `kubectl→forbidden` overrides a specific `kubectl get→allow`. **Two consequences:** (1) you **cannot** build a default-deny allowlist (broad deny + read-only carve-outs) — the deny swallows the carve-outs; the file must stay a **denylist** (enumerate forbidden verbs, leave the rest). (2) An **unmatched** command yields `{"matchedRules":[]}` (no decision) — execpolicy defers to the sandbox, so under `codex exec --sandbox workspace-write -c …network_access=true` (the `codex-mem.sh --executor` mode, no approval prompts) any unmatched command that only needs the network **runs without prompt**. So a denylist must explicitly forbid every mutating verb (`terraform state push/rm/mv`, `terraform import/taint/force-unlock`, `gh api` (raw-REST merge bypass), `kubectl edit/set/drain/exec/run/cp/…`); whole un-enumerated tools (`aws`/`gcloud`/`psql`/`curl`) still run — the real backstop is the no-apply/no-merge instruction restated in every delegation prompt + human gates, with the deny-rules as defense-in-depth. `decision` keyword is `"forbidden"` (not `deny`/`block`).

**[2026-06-15]** `codex-mem.sh --executor-bare` strips the AGENTS.md memory stack (~13.6k tokens) via `-c project_doc_max_bytes=0` — a bare Codex executor lands at ~20.4k tokens, equal to a Claude `Agent` subagent. **Why:** read-only/throwaway subagent work (PR review, render, validate, search) doesn't need the memory stack, and the saving compounds when fanning out many subagents. **How to apply:** use `--executor-bare` as the primary for lean read-only fan-out (Claude `Agent` as fallback); reserve full `--executor` for work that needs project memory context. Deny-rules guardrails still apply in bare mode; the sandbox blocks `rm -rf`, so subagents clean up with a workspace-relative temp dir + `rm -r`.

**[2026-05-17]** OpenAI `codex` stalls silently after sustained use — observed pattern: reliable for the first ~5 tasks of a session, then exits 0 with no output and no diff. Retrying in-place does not help. **Proven fallback:** delegate to a general-purpose Claude subagent with the sonnet model — clean completions for code cutover, template review, and schema work. Rule: when codex stalls, switch agent rather than retry.

**[2026-07-13]** **Codex ships STABLE native hooks — the "Codex has no memory hook" assumption is retired.** `codex features list` shows `hooks stable`, and a `.codex/hooks.json` `UserPromptSubmit` hook injects `additionalContext` the model actually sees — probe-confirmed 2026-07-12 (runtime Codex v0.144.1; the model answered an injected marker). The installed `codex-cli 0.135.0` already carries it — **no upgrade needed**. The Claude and Codex hook contracts are effectively isomorphic (same events, stdin JSON, `additionalContext`, `decision:block`), so Codex can inject memory **live per-prompt** like Claude instead of only rebuilding `AGENTS.md` at launch via `codex-mem.sh`. Consequence: Codex becomes a candidate for mid-session project switching and a native-hook memory path (a hybrid AGENTS.md-base-plus-hooks design), and any doc asserting "Codex has no native memory hook" is now stale.

**[2026-07-14]** **Codex `hooks.json` SCHEMA — and it is NOT Claude's/Antigravity's shape.** Everything nests under a top-level `"hooks"` object keyed by event; each event is an array of groups `{ "matcher"?: "<regex>", "hooks": [ { "type":"command", "command":"<cmd>" } ] }`. `UserPromptSubmit` takes NO matcher; `PreToolUse` takes a tool-name regex matcher (`^Bash$|apply_patch`). Example:
```json
{ "hooks": {
  "UserPromptSubmit": [ { "hooks": [ { "type":"command", "command":"…" } ] } ],
  "PreToolUse":       [ { "matcher":"^Bash$|apply_patch", "hooks":[ { "type":"command", "command":"…" } ] } ] } }
```
Contrast Antigravity's `{ai-memory-inject:{PreInvocation:[{type,command}]}}` (namespaced top-level keys, no wrapping `hooks`, inject has no inner `hooks` array). **A shared registration writer cannot serve both — write a per-harness writer.** Discovery layers (precedence low→high): `$CODEX_HOME/hooks.json`, `$CODEX_HOME/config.toml [hooks]` inline TOML, repo `.codex/hooks.json`/`.codex/config.toml`, then MDM-managed. `CODEX_HOME` overrides `~/.codex` (verified via `codex doctor`).

**[2026-07-14]** **Hook OUTPUT envelopes (probe-verified, codex 0.144.1).** Inject: `{"hookSpecificOutput":{"hookEventName":"UserPromptSubmit","additionalContext":"…"}}` — identical to Claude; the string is injected as developer context. Block a tool: `exit 2` + reason on **stderr** (or `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"…"}}`). **`exit 2` IS honored** — codex logs `codex_core::tools::router: Command blocked by PreToolUse hook: <stderr>` and the model is told the tool was blocked. So an exit-2 shell guard works on Codex.

**[2026-07-14]** **Hook STDIN schemas differ by harness — this is a fail-OPEN trap.** Codex `PreToolUse` stdin puts the shell command at **`tool_input.command`** — NOT Antigravity's `toolCall.args.CommandLine`. Full captured shape: `{session_id,turn_id,transcript_path,cwd,hook_event_name:"PreToolUse",model,permission_mode,tool_name:"Bash",tool_input:{command},tool_use_id}`. `UserPromptSubmit` stdin: `{session_id,turn_id,transcript_path,cwd,hook_event_name,prompt,…}`. **A guard that reads the wrong path extracts empty and ALLOWS everything** (silent fail-open) — a shared guard must read `tool_input.command` first (Codex/Claude family) and fall back to `toolCall.args.CommandLine` (Antigravity), never trusting one path. Always **capture the real stdin** (a dump-to-file hook) before writing a guard; a unit test that hard-codes the wrong schema will pass green while the guard is broken. See [[preventing-drift]] (test-matches-the-bug).

**[2026-07-14]** **`requirements.toml` is MDM/enterprise-ONLY — dead for auto-trusting hooks on a personal install.** Non-managed hooks (`hooks.json`/`config.toml [hooks]`) require a one-time `/hooks` trust (hash-pinned; re-prompts when the hook command/hash changes) or `--dangerously-bypass-hook-trust`. `requirements.toml` marks hooks "managed/trusted-by-policy" — BUT it is delivered through a device/MDM channel (macOS payload `com.openai.codex:requirements_toml_base64`), NOT read from `$CODEX_HOME/requirements.toml`. Probe (auth-fixed, codex 0.144.1): a `$CODEX_HOME/requirements.toml` hook did not fire even WITH `--dangerously-bypass-hook-trust` — i.e. it was never discovered. **So there is no install-writable auto-trust path off MDM.** For a personal install: write user-level `~/.codex/hooks.json`, document a one-time `/hooks` trust for interactive use, and `--dangerously-bypass-hook-trust` on headless `codex exec` (the `codex-mem.sh` executor path). Forging codex's trust-hash store from an installer is rejected (fragile, version-brittle, defeats the security model).

**[2026-07-14]** **Probing codex hooks hermetically — two traps that cause false results.** (1) A hermetic `CODEX_HOME=<tmp>` isolates AUTH too → every `codex exec` fails `401 Unauthorized` and the model never runs, so a hook that DID fire still looks like "not injected." **Seed `cp ~/.codex/auth.json "$CODEX_HOME/"`** (and copy `config.toml` minus `[hooks]/[features]`) before probing. (2) A standalone `codex exec "prompt" -o out` **blocks reading stdin** ("Reading additional input from stdin…") and hangs indefinitely — it needs **`</dev/null`** (this is exactly why `codex-mem.sh` executor mode redirects stdin from `/dev/null`). Both are probe-harness bugs, not codex behavior; both produce convincing-looking wrong answers. General rule: verify tool capability by PROBING the real binary with a marker, and make the probe itself falsifiable. See [[agent-tooling]], [[preventing-drift]].
