**[2026-06-17]** **Project-scoped skills** — a skill that should only fire in one repo lives at `<memory>/projects/<project>/skills/<skill>/` (the path *is* the scope; sits next to that project's `memory.md`/`repo_path`), not in the global `<memory>/skills/`. `scripts/sync-project-skills.sh` fans each into its repo's per-harness skill dir, resolving the project's checkout via the shared `resolve_repo_path` (`repo_path` is **relative** to `AI_MEMORY_PROJECTS_ROOT`, or the `$MEMORY_DIR` sentinel for the self-referential meta-project, git remote as fallback): **Claude Code → `<repo>/.claude/skills/<skill>`, Codex → `<repo>/.agents/skills/<skill>`**. Both harnesses consume the **same `SKILL.md`** (Codex added skills Dec 2025, identical format), so **no translation** is needed — unlike agents. Flags: `--harness claude|codex|all` (default all), `--mode link|copy` (default link), `--force` (refresh a copy), `--list`, `--dry-run`, optional `[<project>…]` filter; idempotent, repairs stale links, refuses to clobber real files / foreign symlinks, skips `SKILL.md`-less dirs (so `*-workspace` dirs are ignored). **link** = personal (symlink points at the absolute store path → machine-local, **breaks for teammates/CI** → gitignore it); **copy** = shareable/committable (Codex repo skills are designed to be checked in). **Precedence gotcha (load-bearing):** Claude Code resolves **Enterprise > Personal (`~/.claude/skills`) > Project (`<repo>/.claude/skills`) > Bundled** — so migrating a skill to project scope **REQUIRES removing its global `~/.claude/skills` symlink**, else the global copy shadows the scoped one everywhere. Prereq: every project `memory.md` carries a resolvable `repo_path` — stored **relative** to `AI_MEMORY_PROJECTS_ROOT` (the `$MEMORY_DIR` sentinel for the self-referential meta-project), resolved to an absolute checkout by `resolve_repo_path`. **Scope a skill by where it should TRIGGER, not its name:** a `<client>-infrastructure-analyzer` skill whose trigger is generic ("any `<client>/<env>` IaC repo") stays **global** despite the client-specific name; the sync mechanism exists for skills that are genuinely single-repo. Default for a real project skill: `--mode link --harness all`. Symlink-following inside `.claude/skills`/`.agents/skills` is undocumented for both harnesses but empirically works for Claude (global links load); verify Codex before relying on `link` there, else use `copy`. Pairs with the [2026-06-16] agents entry below.

**[2026-06-16]** Harness-agnostic **agent** management — same canonical-store-plus-symlink pattern as skills, applied to subagent definitions. **Store:** `<memory>/agents/` holds flat `<name>.md` files (agents are single files, unlike skills which are dirs with a `SKILL.md`). **Linker:** `scripts/link-agents.sh` — a sibling of `link-skills.sh`, not a generalization of it (the two differ in entry shape: file vs dir, and validity gate). Default target `~/.claude/agents`, `[TARGET_DIR]` arg + `AGENTS_SRC` env override, `--list`/`--dry-run`; idempotent, repairs stale links, refuses to clobber real files / foreign symlinks. **Validity gate = non-empty file whose first line is `---`** (has YAML frontmatter) — this is what drops empty stubs (e.g. a 0-byte `platform-engineer.md`) and stray non-agent files. **Cross-provider reuse is symlink-only for now (YAGNI):** the system-prompt body is portable and the frontmatter (`name`/`description`/`tools`/`model`) is currently Claude-Code-shaped; Claude Code is the only consumer. **Extension point:** when a harness with a divergent agent schema becomes a real consumer, add a transform-and-copy branch in the linker for that target (emit a translated file instead of a symlink) — localized to the linker, no store change. Decision to keep a sibling script (vs shared `_lib.sh` or a `--agents` mode on `link-skills.sh`): defer the shared-lib refactor until a third linker exists. Pairs with the [2026-06-14] skills entry below.

**[2026-06-14]** Harness-agnostic skill management — own skills in one canonical store, symlink them into each runtime. **Problem:** "skills" are just a dir with a `SKILL.md`, but each agent harness expects them in its own location (`~/.claude/skills/` for Claude Code, `~/.config/opencode/skills/` for others), so a skill authored for one tool is invisible to the rest and edits drift across copies. **Pattern:** keep the real skill dirs in one source-of-truth store (here: `<memory>/skills/<name>/`), and make each harness's skills dir hold **symlinks** pointing back into it (`~/.claude/skills/<name>` → `<store>/skills/<name>`). One edit, every harness sees it; adding a runtime = re-run the linker against its dir. **Mechanics that matter:** (1) per-skill symlinks, NOT a whole-folder symlink — a folder-level link orphans any skill the harness already had that you didn't migrate; per-entry links coexist with non-managed dirs. (2) Only link dirs that actually contain `SKILL.md` — skill-creator leaves `*-workspace` scratch/eval dirs that are harness-local and must NOT be linked out. (3) Linker must be idempotent, repair stale links (target moved), and refuse to clobber a real dir or a foreign symlink it didn't create. (4) Symlinks are transparent to the harness — Claude Code reads `~/.claude/skills/x/SKILL.md` the same whether it's a real dir or a link, so no harness config changes. Reference impl: `scripts/link-skills.sh [TARGET_DIR]` in the memory system (`--list`/`--dry-run`, `SKILLS_SRC` override). Gotcha: any docs/notes citing the old in-harness path (`~/.claude/skills/x/SKILL.md`) still resolve via the symlink, but the editable file now lives in the store — point edits there.

**[2026-07-09]** **A command-proxy hook can silently replace a tool's output with a compressed summary — never infer state from a proxied read.** With the RTK hook installed, bare `ls` and `git status` are rewritten to `rtk <cmd>`, which returns a token-optimised digest rather than real output: `ls -la` on a directory holding three files printed `(empty)`, and `git status --short` on a clean tree printed `ok`. Read literally, that is indistinguishable from data loss, and a session can start "recovering" files that were never gone. **Verify filesystem state with something the proxy does not intercept** — `find <dir> -maxdepth 1 -type f`, the Glob/Read tools, or `rtk proxy <cmd>` for raw passthrough. The general rule for any output-rewriting hook (token optimisers, log filters, pretty-printers): its output is a *rendering*, not an observation, so never let it be the evidence for a destructive or alarming conclusion. Confirm through a second, unproxied channel first. **Causal claim superseded 2026-07-18 (see the entry below): the `(empty)` was NOT a "compressed digest" — `rtk ls` is outright broken on non-`en_US` locales and returns nothing. The behavioural rule above stands; the "it's summarising by design" explanation was wrong and is worth un-learning, because it framed a fixable bug as intended behaviour.**

**[2026-07-13]** **A `cli:` executor is a one-shot headless agentic run — dispatch it in the background, and normalize its output per-harness.** `executor.sh --run` for a `cli:` executor (codex, antigravity) execs the CLI **synchronously in the foreground**, and it runs a **minutes-long** agentic loop — so a foreground Bash-tool call is killed by the harness timeout mid-run (in Claude, 2 min → SIGTERM / exit 143); dispatch it as a background task (in Claude, `run_in_background: true`). The `subagent` plane (the in-harness Agent tool) has no such timeout. Both CLIs are one-shot self-terminating and non-blocking on stdin (`executor.sh` appends `</dev/null`; `codex-mem.sh` also redirects it). **Output shapes differ:** `codex exec` streams a verbose transcript (header, reasoning, `tokens used`, final message printed twice) — the clean final message is available via `codex exec -o/--output-last-message <FILE>` (or `--json` for JSONL events); `agy -p` prints only the final response. **Read-only enforcement differs too:** codex has a real `--sandbox read-only` flag, antigravity has none and relies on a PreToolUse guard that denies writes when `executor.sh` exports `AI_MEMORY_ROLE=explore`. See [[shell]] on why a verbose stream should not be parsed loosely.

**[2026-07-18]** **`rtk`'s `ls` returns EMPTY on any locale outside `en_US`/`C` — and this is the concrete cause behind the [2026-07-09] "proxy digest" entry above.** `rtk 0.38.0` `ls` exits **0 with empty stdout AND empty stderr** under `nl_NL`, `en_GB`, `de_DE`, `fr_FR`; `en_US` and `C` return correct output. Verified on freshly-created directories in both orders, so it is locale, not caching or ordering — and note the split is `en_US`/`C` vs *everything else*, not English vs non-English (`en_GB` fails). Any harness hook that rewrites bare `ls` → `rtk ls` therefore makes **every** `ls` silently return nothing on a non-US machine. **Fix:** `export LC_ALL=en_US.UTF-8` in `~/.zshenv` — *not* `LC_ALL=C` (byte mode mangles non-ASCII content, which matters if your tree holds em-dashes/arrows), and *not* a repo-local config file, because no shell profile sources those. Escape hatch: `rtk proxy <cmd>` bypasses the filter. **The disambiguation is the load-bearing part — two macOS "phantom filesystem" failures look alike and are opposites.** TCC = **loud** `EPERM` on file *contents* while `ls` still lists fine ([[macos-tcc-full-disk-access]]). rtk-locale = **silent** empty listing, exit 0, while reads work fine. Diagnostic: `ls` empty but `find`/`git ls-files` disagree → tooling; `ls` fine but `cat`/`git` `EPERM` → TCC. Getting this backwards cost weeks of misattribution and an entire memory-tree relocation that fixed nothing, because a failure that is **both silent and exit-0** never presents as an error to debug — only as a fact to believe. Corollary for any tool that filters command output: treat "empty" as *unproven*, not as evidence of absence, and confirm via an unproxied channel before concluding anything.
