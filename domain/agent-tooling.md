**[2026-06-17]** **Project-scoped skills** — a skill that should only fire in one repo lives at `<memory>/projects/<project>/skills/<skill>/` (the path *is* the scope; sits next to that project's `memory.md`/`repo_path`), not in the global `<memory>/skills/`. `scripts/sync-project-skills.sh` fans each into its repo's per-harness skill dir, resolving the project's **absolute** `repo_path` from `memory.md`: **Claude Code → `<repo>/.claude/skills/<skill>`, Codex → `<repo>/.agents/skills/<skill>`**. Both harnesses consume the **same `SKILL.md`** (Codex added skills Dec 2025, identical format), so **no translation** is needed — unlike agents. Flags: `--harness claude|codex|all` (default all), `--mode link|copy` (default link), `--force` (refresh a copy), `--list`, `--dry-run`, optional `[<project>…]` filter; idempotent, repairs stale links, refuses to clobber real files / foreign symlinks, skips `SKILL.md`-less dirs (so `*-workspace` dirs are ignored). **link** = personal (symlink points at the absolute store path → machine-local, **breaks for teammates/CI** → gitignore it); **copy** = shareable/committable (Codex repo skills are designed to be checked in). **Precedence gotcha (load-bearing):** Claude Code resolves **Enterprise > Personal (`~/.claude/skills`) > Project (`<repo>/.claude/skills`) > Bundled** — so migrating a skill to project scope **REQUIRES removing its global `~/.claude/skills` symlink**, else the global copy shadows the scoped one everywhere. Prereq: `repo_path` must be absolute in every project `memory.md` (normalized 2026-06-17; relative paths were under `~/Downloads/personal/`). **Scope a skill by where it should TRIGGER, not its name:** `fiter-infrastructure-analyzer` has a generic trigger ("any `<client>/<env>` IaC repo") so it stays **global** despite the fiter name; the sync mechanism exists for skills that are genuinely single-repo. Default for a real project skill: `--mode link --harness all`. Symlink-following inside `.claude/skills`/`.agents/skills` is undocumented for both harnesses but empirically works for Claude (global links load); verify Codex before relying on `link` there, else use `copy`. Pairs with the [2026-06-16] agents entry below.

**[2026-06-16]** Harness-agnostic **agent** management — same canonical-store-plus-symlink pattern as skills, applied to subagent definitions. **Store:** `<memory>/agents/` holds flat `<name>.md` files (agents are single files, unlike skills which are dirs with a `SKILL.md`). **Linker:** `scripts/link-agents.sh` — a sibling of `link-skills.sh`, not a generalization of it (the two differ in entry shape: file vs dir, and validity gate). Default target `~/.claude/agents`, `[TARGET_DIR]` arg + `AGENTS_SRC` env override, `--list`/`--dry-run`; idempotent, repairs stale links, refuses to clobber real files / foreign symlinks. **Validity gate = non-empty file whose first line is `---`** (has YAML frontmatter) — this is what drops empty stubs (e.g. a 0-byte `platform-engineer.md`) and stray non-agent files. **Cross-provider reuse is symlink-only for now (YAGNI):** the system-prompt body is portable and the frontmatter (`name`/`description`/`tools`/`model`) is currently Claude-Code-shaped; Claude Code is the only consumer. **Extension point:** when a harness with a divergent agent schema becomes a real consumer, add a transform-and-copy branch in the linker for that target (emit a translated file instead of a symlink) — localized to the linker, no store change. Decision to keep a sibling script (vs shared `_lib.sh` or a `--agents` mode on `link-skills.sh`): defer the shared-lib refactor until a third linker exists. Pairs with the [2026-06-14] skills entry below.

**[2026-06-14]** Harness-agnostic skill management — own skills in one canonical store, symlink them into each runtime. **Problem:** "skills" are just a dir with a `SKILL.md`, but each agent harness expects them in its own location (`~/.claude/skills/` for Claude Code, `~/.config/opencode/skills/` for others), so a skill authored for one tool is invisible to the rest and edits drift across copies. **Pattern:** keep the real skill dirs in one source-of-truth store (here: `<memory>/skills/<name>/`), and make each harness's skills dir hold **symlinks** pointing back into it (`~/.claude/skills/<name>` → `<store>/skills/<name>`). One edit, every harness sees it; adding a runtime = re-run the linker against its dir. **Mechanics that matter:** (1) per-skill symlinks, NOT a whole-folder symlink — a folder-level link orphans any skill the harness already had that you didn't migrate; per-entry links coexist with non-managed dirs. (2) Only link dirs that actually contain `SKILL.md` — skill-creator leaves `*-workspace` scratch/eval dirs that are harness-local and must NOT be linked out. (3) Linker must be idempotent, repair stale links (target moved), and refuse to clobber a real dir or a foreign symlink it didn't create. (4) Symlinks are transparent to the harness — Claude Code reads `~/.claude/skills/x/SKILL.md` the same whether it's a real dir or a link, so no harness config changes. Reference impl: `scripts/link-skills.sh [TARGET_DIR]` in the memory system (`--list`/`--dry-run`, `SKILLS_SRC` override). Gotcha: any docs/notes citing the old in-harness path (`~/.claude/skills/x/SKILL.md`) still resolve via the symlink, but the editable file now lives in the store — point edits there.
